# Case Study: When Mobile Builders Cannot Ship

Mobile used to attract builders who modified the platform itself. Custom ROMs,
experimental Android distributions, alternative interface ideas, and new mobile
operating systems gave ambitious developers a place to work close to the device.
The feedback loop was imperfect, but the imagination was wide: change the
platform, flash the device, share the build, and see what users did.

That loop narrowed. App stores, signing chains, device locks, review processes,
payments, banking-app compatibility, malware defenses, and hardware support
turned distribution into the binding constraint. A small team can still write an
app, launcher, marketplace, ROM, or operating system layer. Getting it onto
ordinary user devices, keeping it updated, preserving trust, and supporting
enough hardware is much harder.

Live alternatives prove both demand and friction. Independent Android app
repositories can distribute open apps, but installation and updates still run
through platform prompts and policy pressure. Custom Android systems can be
excellent, but they often narrow to specific devices for security and support.
European regulation has opened alternative iOS distribution paths, but those
paths still include authorization, notarization, regional limits, settings flows,
and fees.

The build side is getting cheaper. A smaller team can now generate code, test
compatibility layers, write tooling, and maintain surface area that would have
been too expensive before. But cheaper building exposes the non-code constraint:
getting software onto devices and keeping it there.

The question is where an ambitious mobile builder should intervene. Is the
opening a new mobile operating system, a better ROM, a sideloading helper, an app
store, developer tooling, device support, payment flow, policy fight, or some
combination that changes the user path?
