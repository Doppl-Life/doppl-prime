# Case Study: When The Old Advice Was To Stay Small

For years, a common software warning was to avoid boiling the ocean. Do not
rebuild the package registry. Do not rethink source control. Do not build a new
team communications system. Do not create a new mobile distribution layer. Do
not make your own benchmark ecosystem. Pick a narrow wedge, ship the smallest
thing, and avoid infrastructure traps.

That warning was often correct. The hard part of many systems was not the first
prototype. It was the long tail: package metadata, weird Git states, sync
conflicts, device policies, user trust, security reviews, migration paths,
documentation, support, and compatibility with old workflows. Big rewrites
failed because coordination and maintenance cost overwhelmed the insight.

The constraint may be moving. AI agents can generate code, inspect large diffs,
run checks, produce migration scaffolds, research prior art, maintain repeated
tasks, and operate disposable environments. They do not remove complexity, but
they can lower the cost of attacking systems that were previously too broad for
small teams.

That does not mean every large idea is newly good. Some oceans are still oceans.
The bottleneck may move from code to trust, distribution, governance,
regulation, data rights, human attention, or who owns a shared state transition.
The old advice becomes dangerous only when it hides a changed constraint.

The question is what should be reconsidered now. Which formerly absurd systems
problems are newly approachable because implementation and coordination cost
fell? Which are still blocked by non-code constraints? How should a builder tell
the difference between a once-impossible project whose bottleneck moved and a
classic overreach with better tools?

## Synopsis

Conventional software advice warned against rebuilding infrastructure (do not boil the ocean); the case probes whether that caution still holds once the hard part of these systems has shifted.
