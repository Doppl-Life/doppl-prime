# /tdd brief — ollama_provider_adapter_behind_modelgateway

## Feature
Add a **local-provider (ollama) adapter behind the `ModelGateway`** (safety rule #9 — vendor/HTTP confined to a client seam, never in domain/runtime) and make `createLiveGateway` **dispatch by `route.provider`** so a request whose resolved route's `provider === 'ollama'` is served by the new adapter (today the live gateway is OpenRouter-only → an ollama route would be dead code). The adapter mirrors the OpenRouter pattern: raw output (the gateway shell owns validate/repair(≤1)/reject — not re-implemented), `withRetry` (bounded retries + per-role timeout + one fallback), structured-output via JSON mode + a **candidate-INDEPENDENT** system instruction (rule #5 isolation, LESSON lesson 98), and a terminal `ProviderCallError` with **zero tokens** on failure (rule #8). ollama is **keyless** (local `OLLAMA_BASE_URL` config, no API key — rule #4 no-leak holds trivially). This is the runtime honoring of FB.0's contract surface; FB.2 layers the per-run `modelRouteOverride` clamp on top.

## Acceptance criteria (what "done" means)
- [ ] **`OllamaClient` seam (rule #9):** a new `apps/api/src/model-gateway/adapters/ollama.adapter.ts` defines an `OllamaClient` interface + `createOllamaClient(env)` factory that is the ONLY place the ollama HTTP/SDK call lives; the module exports NO vendor type and imports no SDK into domain/runtime. The factory reads `OLLAMA_BASE_URL` (default `http://localhost:11434`) — **no API key** (keyless local; rule #4 holds by construction).
- [ ] **`createOllamaProviderCall(deps)` returns a `ProviderCallFn`** mirroring OpenRouter: returns **raw** output + `providerMeta` (the gateway shell validates — adapter does NOT); uses `withRetry` (bounded retries default 2 + per-role timeout + one fallback); a structured request (`capability.structuredOutputs && request.schema`) sets ollama JSON mode + prepends a **candidate-INDEPENDENT** system instruction derived from `z.toJSONSchema(request.schema)` (rule #5 / lesson 38 isolation — never candidate text); a terminal failure throws `ProviderCallError(failures, providerMeta)` with **`tokensIn:0, tokensOut:0`** (rule #8 — no energy on failure).
- [ ] **`providerMeta` carries `provider:'ollama'` + `modelId` + `gatewayRequestId` + token counts; NO key/secret field** anywhere in the output, `providerMeta`, or the error (rule #4 — asserted by a no-leak test; trivially true since ollama is keyless, but pinned).
- [ ] **`createLiveGateway` dispatches by `route.provider`:** the injected `providerCall` resolves the request's route (`registry.resolve(role)`), reads `route.provider`, and dispatches to the matching adapter's `providerCall` — `'openrouter'` → the existing adapter, `'ollama'` → the new one. An **unregistered provider** → an honest `ProviderCallError` (no silent fallback — this dispatch map IS the runtime provider allowlist). The embedding role/path is **unchanged** (not regressed — see Step-2.5 Q1).
- [ ] **Reachable (Step 7.5):** a `ModelGatewayRequest` whose resolved route's `provider === 'ollama'` is served end-to-end by the ollama adapter through `createLiveGateway` → `createGateway`'s validate/repair/reject (proven by an integration-style test driving the real composition with a faked `OllamaClient`). Boot (`main.ts`) constructs the keyless ollama client alongside the OpenRouter client when live.
- [ ] **Replay isolation (rule #7):** the ollama adapter is never invoked on the replay path; the recorded/fake gateway (`selectGateway` stub branch) is unaffected (replay reads the persisted route + outcome, no provider call).
- [ ] **Capability:** an ollama route's `ProviderCapability` is `{structuredOutputs:true, embeddings:false}` for FB.1 (embedding stays OpenAI-pinned — Step-2.5 Q2).
- [ ] All model-gateway tests pass; `/preflight` clean.
- [ ] **security-reviewer (INVARIANT):** FB.1 touches the secret boundary (rule #4) + SDK-isolation (rule #9) → run security-reviewer at Step 8 (the no-leak + no-SDK-in-domain assertions are the slice's safety core).

## Things to flag at Step 2.5
1. **Provider-dispatch shape + the embedding-role interaction.** `createLiveGateway` becomes provider-aware. My default vote: build a **`provider → providerCall` map** inside `createLiveGateway`; the injected `providerCall` resolves `route.provider` per request and dispatches; an unregistered provider → honest `ProviderCallError`. **Critical:** today the live gateway routes the embedding role through the OpenRouter call too (a latent gap — the demo uses the recorded gateway). FB.1 must NOT make an `embedding`/`openai` route throw — either scope the dispatch to the generation providers and leave the embedding role on its current (recorded-only) path, OR register a passthrough. Default: **scope the dispatch map to the generation providers `{openrouter, ollama}`; leave embedding exactly as today (out of FB.1 scope, noted)** — flag if the implementation reveals embedding is actually exercised live.
2. **ollama embedding adapter — in scope or defer?** My default vote: **DEFER.** Embeddings stay OpenAI-pinned (`text-embedding-3-small`) — novelty quality + vector-dimension stability are demo-critical (rule #7 persists vectors), and a mixed-dimension corpus would corrupt cosine novelty. ollama route capability = `{structuredOutputs:true, embeddings:false}`. A separate slice can add ollama embeddings if wanted.
3. **Keyless boot vs `REQUIRED_CREDENTIAL_ENV`.** My default vote: **ollama client is keyless** (`OLLAMA_BASE_URL` only, default localhost:11434, lazy) and FB.1 does **NOT** touch `REQUIRED_CREDENTIAL_ENV` — the demo's default routes still need OpenRouter/OpenAI keys, so the boot gate stays. A fully key-free all-ollama boot (relaxing the gate when no OpenRouter/OpenAI route is configured) is a separate relaxation, OUT of FB.1 scope (note it).
4. **ollama transport: raw HTTP `fetch` vs the official `ollama` JS SDK.** My default vote: **raw HTTP `fetch`** to the ollama REST API (`/api/chat` or `/api/generate` with `format:'json'`) behind the `OllamaClient` seam — no new dependency, the REST surface is small/stable, and rule #9 is satisfied either way (SDK/HTTP confined to the factory). Confirm the exact endpoint/shape against current ollama docs (Context7) at GREEN.
5. **JSON/structured-output mode.** My default vote: mirror OpenRouter — request ollama's JSON mode (`format:'json'`) + the candidate-independent schema instruction; the gateway's Zod validate/repair/reject stays the authoritative validator (rule #5 UNWEAKENED, exactly as LESSON lesson 98 for OpenRouter).
