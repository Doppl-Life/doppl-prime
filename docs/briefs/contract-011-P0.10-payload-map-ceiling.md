# /tdd brief — per_type_payload_map_and_ceiling

## Feature
Add a **per-type payload-shape narrowing layer** over the generic `RunEventEnvelope.payload` (JSONB) for the six high-traffic event types named in §4, plus a **payload size/depth ceiling** (the P0.1-security carry-forward). The narrowing maps each high-traffic event `type` to the corresponding frozen Appendix-A model — `energy.spent`←`EnergyEvent`, `candidate.created`←`CandidateIdea`, `critic.reviewed`←`CriticReview`, `check.completed`←`CheckResult`, `novelty.scored`←`NoveltyScore`, `fitness.scored`←`FitnessScore` — so the SAME Zod schema validates the event-store write and the model. A high-traffic event whose payload doesn't match its narrowed shape is **rejected**; any type outside the high-traffic set falls back to the generic JSONB payload. The ceiling (`enforcePayloadCeiling`) is a pure, **bounded** check (serialized size + nesting depth) that the P1 append path also calls. **SECURITY slice** (input-validation on the authoritative log + payload-DoS defense; consumes the P0.1 security carry-forward). SOLO — own commit, never bundled with feature work.

## Acceptance criteria (what "done" means)
- [ ] `HIGH_TRAFFIC_PAYLOAD_MAP` maps EXACTLY the six high-traffic `RunEventType` literals to their model schema: `energy.spent`→`EnergyEvent`, `candidate.created`→`CandidateIdea`, `critic.reviewed`→`CriticReview`, `check.completed`→`CheckResult`, `novelty.scored`→`NoveltyScore`, `fitness.scored`→`FitnessScore`. Keys are statically constrained to `RunEventType` members (Q2).
- [ ] `resolvePayloadSchema(type)` returns the **narrowed** schema for a high-traffic type and the **generic** payload schema (`z.record(z.string(), z.unknown())`, matching the envelope payload shape) for any other `RunEventType` member. Lookup is **own-property** (defeats `__proto__`/`constructor`/`toString` map-key bypass — lesson §11), so a prototype-key `type` resolves to generic, never a borrowed schema.
- [ ] A high-traffic event whose payload does NOT match its narrowed shape is **rejected** (e.g. an `energy.spent` payload missing `actual`, a `critic.reviewed` payload carrying a `winner` field). A type outside the high-traffic set validates against the generic JSONB payload (accepted) — §4 fallback.
- [ ] `novelty.scored` narrows to `NoveltyScore`, which already carries the persisted `vector` + `embeddingModelId` + `dimension` (§9 authoritative-once-computed) — so the narrowed payload structurally requires them.
- [ ] `fitness.scored` narrows to `FitnessScore`, which references the novelty it consumed via the shared `candidateId` + the `components` novelty signal (§8 explainability) — see Q1 for the explicit-FK option.
- [ ] **Payload ceiling (security carry-forward):** `enforcePayloadCeiling(payload)` returns `{ ok: true }` when serialized size ≤ `MAX_PAYLOAD_BYTES` AND nesting depth ≤ `MAX_PAYLOAD_DEPTH`, else `{ ok: false, violation: 'max_bytes' | 'max_depth' }`. The depth check is **bounded on attacker input** — it early-exits the instant depth exceeds the limit and never fully traverses / stack-overflows on a pathological deeply-nested object (Q5). `MAX_PAYLOAD_BYTES` / `MAX_PAYLOAD_DEPTH` are exported constants (Q3).
- [ ] `validateEventPayload(type, payload)` composes the two: enforces the ceiling, then validates against the resolved (narrowed-or-generic) schema; a ceiling violation OR a high-traffic shape mismatch fails (Q7 for ship-now-vs-P1).
- [ ] **Schema-snapshot test (§2.5 gate, tagged `spec(§4)`):** the high-traffic key-set (the six literals) is frozen, every key is a valid `RunEventType` member, and each maps to its expected model schema (a per-key fixture round-trips through `resolvePayloadSchema`). Adding/removing/remapping a high-traffic type is caught as a cross-track regression.
- [ ] `z.infer` types + all new symbols re-exported from the `@doppl/contracts` barrel; all unit tests pass; `/preflight` clean (use the **package-pinned** prettier, not bare `npx` — lesson §14).

## Things to flag at Step 2.5
1. **`fitness.scored` ↔ novelty link — reuse `FitnessScore` as-is (A) vs extend the payload with an explicit `noveltyScoreId` (B).** My default vote: **A — reuse `FitnessScore` directly.** §4 says "reuse the corresponding Appendix-A model … so the same schema validates the write and the model"; the §8 "references the novelty it consumed" link is the shared `candidateId` + the `components.novelty` signal (exactly one `novelty.scored` per candidate in MVP). B (a required `noveltyScoreId` on the payload only, leaving the frozen `FitnessScore` at 6 fields) makes the FK structural but adds a field the model doesn't have — pick B only if a hard event-to-event FK is wanted now over the candidateId join (P0.8 deferred this link "to the event-payload layer (P0.10)", so B is defensible). I lean A (YAGNI, lesson §6).
2. **Map key typing.** My default vote: type the map as `Partial<Record<RunEventType, z.ZodType>>` so a typo'd or non-member key is a COMPILE error, AND snapshot the key-set (#13). Flag if you'd rather a plain `Record<string, …>` + only the runtime snapshot.
3. **Ceiling constant values.** My default vote: `MAX_PAYLOAD_BYTES = 1_048_576` (1 MiB) and `MAX_PAYLOAD_DEPTH = 32`. Rationale: a realistic `novelty.scored` 1536-float vector serializes to ~30 KB (well under 1 MiB), and 32 nesting levels exceeds any legitimate Appendix-A payload (max ~3–4 deep). These are MVP DoS bounds, tunable; the kernel/event-store (P1) may re-pass stricter values. Flag if you want different MVP numbers.
4. **Byte measurement.** My default vote: `JSON.stringify(payload).length` (UTF-16 code units — pure, portable, no `Buffer` dependency; a conservative over-count for a DoS ceiling, where over-restricting is safe). Flag if you require true UTF-8 byte count (`Buffer.byteLength(JSON.stringify(payload),'utf8')` — still pure, a Node global). Note: a payload with a `BigInt`/circular value would throw in `JSON.stringify` — payloads are plain JSON objects by contract, but consider catching and treating an unserializable payload as a `max_bytes`-style violation rather than letting it throw.
5. **Depth-check must be bounded on attacker input.** My default vote: an **explicit-stack iterative traversal** (or recursion that returns the violation the instant `depth > MAX_PAYLOAD_DEPTH`, never descending further) so a depth-100k object yields `{ok:false, violation:'max_depth'}` and never throws a stack overflow. This is the load-bearing security property (test #10). Confirm the implementation can't be tricked into full traversal before the early-exit.
6. **`enforcePayloadCeiling` signal — result object vs throw.** My default vote: **result object** `{ ok, violation? }` (rejection is control flow, not an exception — mirrors `resolveCheckAdapter` + the worked-example caps pattern). The P1 append path emits a violation event on `ok:false`. Flag if you'd rather throw.
7. **Ship the composed `validateEventPayload` now vs leave composition to P1.** My default vote: **ship it now** — the contract owns the full narrow+ceiling entry; P1 just calls it and decides the event emit. Flag if you'd rather export only the primitives (map + resolver + ceiling) and compose at P1.
8. **Commit count.** My default vote: **1 — SOLO security slice** (input-validation on the authoritative event log + payload-DoS ceiling; consumes the P0.1 security carry-forward). Never bundled. Commit: `feat(contracts): per-type payload map + payload ceiling (P0.10)`.
