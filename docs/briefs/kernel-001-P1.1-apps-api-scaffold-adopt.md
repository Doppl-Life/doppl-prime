# /tdd brief — apps_api_scaffold_and_contracts_adopt

## Feature
Stand up the `@doppl/api` workspace package (build scaffolding — `package.json`, `tsconfig`, `vitest`, scripts) and **adopt** the frozen `@doppl/contracts` from the kernel side: a smoke test proves the workspace package seam resolves into `apps/api` and the adopted Zod contracts parse. This is the prerequisite slice-0 of the freeze bundle — every subsequent P1/P2 slice needs `apps/api` to build + run Vitest.

## Use case + traceability
- **Task ID:** P1.1 (kernel-side **adopt** of the §4 event-model contracts — this slice consumes, it does NOT re-author them)
- **Architecture sections it implements:** `ARCHITECTURE.md §4` (the event-model contracts being adopted)
- **Related context:** the §4 event-model contracts were authored + snapshot-pinned in Phase 0 (the contract track — RunEventEnvelope, RunEventType, EvidenceRef, the per-type payload map, and the P0.14 contract-test surface), so this slice does not redefine any of them. Phase 0 froze `@doppl/contracts` (36 exports, schemaVersion 2) at `f34094d`. `apps/api` is currently bare (only `CLAUDE.md` + `LESSONS.md`) — no package manifest exists. Mirror the existing `packages/contracts` scaffolding (`package.json` / `tsconfig.json` / `vitest.config.ts`) for consistency. This is infra scaffolding carrying one adopt smoke test as its RED→GREEN anchor — keep it minimal; the P1/P2 feature slices add their own runtime deps (Drizzle in P1.4, pg in P1.3, provider SDKs in P2.5/2.6) so each dependency is traceable to the slice that introduces it.

## RED test outline (Step 2)
Tests to write in `apps/api/test/unit/scaffold.smoke.test.ts`:

1. **`test_contracts_workspace_seam_resolves`** — the `@doppl/contracts` barrel resolves into `apps/api`.
   - Asserts: `import { CURRENT_SCHEMA_VERSION } from '@doppl/contracts'` is a number `>= 1` (do NOT hard-code `=== 2` — the version is allowed to bump and this smoke test must not pin it).
   - Why: pins the workspace package seam + Vitest alias + tsconfig module resolution all work end-to-end (P1.1 adopt).

2. **`test_adopted_contract_schema_parses`** — an adopted Zod contract is usable (not merely type-resolvable) from `apps/api`.
   - Asserts: `RunEventEnvelope.safeParse(validRunEventEnvelope).success === true` (both imported from `@doppl/contracts`, the fixture from the P0.14 test-fixtures surface).
   - Why: proves the adopted §4 event-model contract validates at runtime from the consuming package, confirming the freeze is truly consumable downstream.

## Things to flag at Step 2.5
1. **Local `eslint.config` for `apps/api`, or rely on the root flat config?** `packages/contracts` ships NO local eslint config and `eslint .` resolves the root `eslint.config.mjs` (it was green through P0 preflight). My default vote: **mirror contracts — no local config**; only add one if `eslint .` from `apps/api` fails to discover the root config (flag if so).
2. **`test:integration` with no integration files yet.** `vitest run test/integration` errors on "no test files found." My default vote: **add `--passWithNoTests`** to `test` / `test:unit` / `test:integration` so the empty integration dir passes until P1.3/P1.4 populate it; the later integration slices then have a working script. Alternative: create a `.gitkeep`'d `test/integration/` with a trivial placeholder — I prefer `--passWithNoTests` (no throwaway test).
3. **Runtime dep set in this slice.** My default vote: **contracts + zod only** (plus dev tooling); defer `drizzle-orm`/`pg`/`postgres`/`fastify`/provider SDKs to the slices that first import them, so each dep addition is bisectable to its consumer. Flag if you'd rather front-load them.
4. **`src/index.ts` placeholder content.** My default vote: a single named export marking the package boundary (e.g. `export const DOPPL_API_PACKAGE = '@doppl/api';`) so `typecheck`/`noUnusedLocals` are satisfied and `src` is non-empty — replaced by real barrels as P1/P2 land.
