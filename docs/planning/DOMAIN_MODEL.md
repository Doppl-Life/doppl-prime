# Doppl Domain Model

Source PRD: `Doppl_Capstone_Proposal.pdf`

## Phase 1 - Product Mechanics

### Core Object Of Value

[locked decision] The canonical unit of work is a **candidate idea** generated by an agenome during a run.

[locked decision] The MVP supports both PRD prey types as candidate idea subtypes:

- `cross_domain_transfer`: an idea that maps a technique, result, or pattern from source domain A onto a target problem in domain B.
- `zeitgeist_synthesis`: a thesis, product concept, or framing that fits the present moment and survives adversarial scrutiny.

[proposed recommendation] The architecture should model candidate ideas as one shared domain concept with subtype-specific fields and checks, rather than building two unrelated products.

### Shared Lifecycle

[locked decision] Both candidate idea subtypes use one shared evolution lifecycle:

1. Seed a run with a prompt or fixed problem set.
2. Spawn a bounded population of agenomes.
3. Each agenome produces one or more candidate ideas.
4. Candidate ideas are normalized into the shared candidate idea schema.
5. The critic council evaluates each candidate idea.
6. Subtype-specific checks add evidence:
   - `cross_domain_transfer`: source-domain validity, target-problem fit, mapping quality, prior-art search, and executable / toy checks where feasible.
   - `zeitgeist_synthesis`: grounding in current signals, novelty, audience / market timing, internal coherence, falsifiability, and held-out judgment where feasible.
7. Fitness is computed from critic scores, subtype checks, novelty / diversity signals, and energy efficiency.
8. Weak lineages are culled.
9. Strong parent agenomes fuse and mutate into child agenomes.
10. The next generation repeats the lifecycle.
11. The run ends by budget, generation count, manual stop, or demo stop condition.

### State-Changing Actions

[locked decision] State changes include run creation, generation start, agenome spawn, energy spend, candidate idea creation, critic review completion, objective / subtype check completion, fitness computation, culling, fusion, mutation, generation completion, and run completion.

### Lifecycle Success / Failure States

[locked decision] A successful run produces at least one surviving candidate idea with traceable lineage, critic evidence, subtype-specific verification evidence, and an improved generation-over-generation fitness signal.

[locked decision] A failed run may end because all candidates are culled, energy is exhausted, objective checks fail, critics reject all candidates, provider/tool failures exceed retry limits, or the demo operator manually stops execution.

### Hidden Mechanics

[production-hardening] Even under MVP posture, energy accounting and termination controls are correctness requirements. The system cannot depend on human restraint to prevent recursive runaway.

[proposed recommendation] Fitness should not be a single unexplainable number. The score should preserve critic dimensions and subtype-check evidence so reviewers can understand why a lineage survived.

### Early Domain Nouns

[locked decision] `Agenome`: serialized agent genome containing system prompt, persona / value weights, tool permissions, decomposition policy, and spawn budget.

[locked decision] `Run`: bounded execution started from a seed prompt or fixed problem set.

[locked decision] `Generation`: one iteration over a population of agenomes and their candidate outputs.

[locked decision] `Population`: the active set of agenomes in a generation.

[locked decision] `Energy`: finite compute/token/tool budget available to an agenome.

[locked decision] `Candidate idea`: the output being evaluated, selected, fused, mutated, and improved.

[locked decision] `Critic council`: adversarial verifier panel evaluating candidate ideas against distinct mandates.

[locked decision] `Objective check`: a domain-specific check outside the breeding loop that can support or reject a candidate idea.

[locked decision] `Fitness score`: structured score derived from critic judgments, objective checks, novelty/diversity signals, and possibly held-out judgment.

[locked decision] `Lineage`: parent/child ancestry connecting agenomes, candidate ideas, fusion events, mutations, and survival/culling events.

### Units / Scores / Records

[locked decision] Unit of work: candidate idea.

[locked decision] Unit of value: verified improvement in candidate idea quality across generations.

[locked decision] Primary score: fitness score, decomposed into auditable components.

[locked decision] Primary records: run, generation, agenome, candidate idea, critic review, objective check result, energy event, reproduction event, lineage event, and dashboard event.

### Confirmed Mechanics

[locked decision] Candidate ideas are the unit of work.

[locked decision] Cross-domain transfer and zeitgeist synthesis are both in MVP scope.

[locked decision] Both idea types share one evolution lifecycle with subtype-specific checks.

### Still Ambiguous

[open question] The exact scoring weights across critic scores, subtype checks, novelty / diversity, and energy efficiency are not yet locked.

## Phase 5 - Domain Model And State Machines

### Source Of Truth

[locked decision] The authoritative source of truth for a run is an append-only run event log.

[locked decision] Current run state, dashboard projections, summary tables, and replay views are derived from the event log.

[locked decision] Historical run events cannot be edited in place. If the system needs annotations or corrections, those must be represented as new events or separate non-authoritative annotations.

### Core Entities

| Entity | Definition | Key Fields | Source Of Truth |
|---|---|---|---|
| Run | Bounded execution of Doppl from a seed prompt/problem set through one or more generations | id, seed, enabled idea types, caps, status, started_at, completed_at | Event log plus run projection |
| Generation | Iteration over a population of agenomes inside a run | id, run_id, index, status, started_at, completed_at | Event log plus generation projection |
| Agenome | Serialized agent genome that can generate candidate ideas | id, run_id, generation_id, parent_ids, system_prompt, persona/value weights, tool permissions, decomposition policy, spawn budget, mutation metadata | Event log plus agenome projection |
| CandidateIdea | Unit of work and value being evolved | id, run_id, generation_id, agenome_id, subtype, title, summary, claims, evidence refs, status | Event log plus candidate projection |
| CriticReview | Structured adversarial review of a candidate idea | id, candidate_id, critic mandate, score dimensions, critique, confidence, evidence refs | Event log |
| CheckResult | Subtype-specific validation result | id, candidate_id, check_type, status, score, output, evidence refs, error | Event log |
| FitnessScore | Decomposed score used for selection | id, candidate_id, total, components, policy version, explanation | Event log |
| NoveltyScore | Simple semantic-distance signal used to reduce mode collapse | id, candidate_id, embedding_ref or method, comparison_set, score, explanation | Event log or fitness component |
| EnergyEvent | Metered compute/tool/spawn spend | id, run_id, generation_id, agenome_id, event_type, amount, provider metadata | Event log |
| ReproductionEvent | Fusion, crossover, output synthesis, or mutation event producing child agenomes | id, run_id, parent_agenome_ids, child_agenome_id, method, mutation summary | Event log |
| CullingEvent | Selection event removing a lineage/candidate from survival | id, run_id, generation_id, target ids, reason, score snapshot | Event log |
| DashboardProjection | Read model optimized for live/replay UI | run id, tree nodes, status, timeline, charts, selected candidate | Derived view |
| LineageGraphProjection | Storage-agnostic nodes/edges projection for dashboard and analysis | run id, nodes, edges, lineage metrics, backing store metadata | Derived view |

### Relationships

[locked decision] A run has many generations.

[locked decision] A generation has many agenomes.

[locked decision] An agenome may have zero, one, or two parent agenomes. Fusion offspring should usually have two parents; seed agenomes have none.

[locked decision] An agenome creates zero or more candidate ideas.

[locked decision] A candidate idea has one subtype: `cross_domain_transfer` or `zeitgeist_synthesis`.

[locked decision] A candidate idea has many critic reviews and zero or more subtype check results.

[locked decision] A candidate idea has one selected fitness score per scoring policy version used for selection.

[locked decision] Candidate ideas in the MVP should receive a simple novelty score when embeddings or equivalent semantic comparison are available.

[locked decision] Energy events attach to runs, generations, and usually agenomes.

[locked decision] Reproduction and culling events define lineage changes, not mutable parent/child rewrites.

[locked decision] Lineage graph consumers should depend on a graph projection contract, not on the physical storage engine.

### State Machines

#### Run State

```text
configured -> running -> completing -> completed
configured -> running -> stopping -> stopped
configured -> running -> failed
configured -> cancelled
```

[locked decision] Terminal run states: `completed`, `stopped`, `failed`, `cancelled`.

[locked decision] A run cannot transition out of a terminal state.

#### Generation State

```text
pending -> running -> verifying -> scoring -> reproducing -> completed
pending -> running -> failed
pending -> skipped
```

[locked decision] A generation can create a next generation only from `completed` and only if run caps allow it.

#### Candidate Idea State

```text
created -> under_review -> checked -> scored -> selected
created -> under_review -> rejected
created -> invalid
scored -> culled
```

[locked decision] A candidate cannot be selected for parent contribution until it has a fitness score.

#### Agenome State

```text
seeded -> active -> spent -> eligible_parent
seeded -> active -> failed
eligible_parent -> reproduced
eligible_parent -> culled
```

[locked decision] An agenome cannot spend energy after it is `spent`, `failed`, or `culled`.

### Business Rules

[locked decision] All selection decisions must be explainable from persisted event data.

[locked decision] Fitness computation must preserve component scores and policy version.

[locked decision] The system must be able to replay a run from events without calling LLM providers again.

[locked decision] Cross-domain transfer and zeitgeist synthesis share lifecycle states but use subtype-specific check result types.

[locked decision] Run caps include at least population size, generation count, energy/token budget, spawn depth, tool call count, and wall-clock timeout.

[locked decision] The best surviving idea must be traceable back to its agenome, parents, generation, critic reviews, checks, energy spend, and score.

[locked decision] Visible lineage specialization should be represented through persisted traits, mutation summaries, score patterns, critic feedback patterns, and/or dashboard labels derived from those facts.

### Invariants

[locked decision] Energy spend cannot exceed assigned run/agenome caps.

[locked decision] Recursive spawning cannot exceed configured depth and population caps.

[locked decision] Critic agents cannot directly mutate candidates, lineage, fitness policy, or survival records.

[locked decision] Check runners cannot directly select winners; they only emit structured evidence.

[locked decision] Dashboard projections are never authoritative.

[locked decision] The event log is append-only for authoritative run facts.

[locked decision] Graph database projections, if introduced, are derived read models and never replace the authoritative event log.

[locked decision] Replay uses persisted events and stored outputs, not fresh model calls.

[locked decision] A child agenome must record parentage and mutation/fusion metadata.

[locked decision] A selected final idea must have critic evidence and a fitness explanation.

### Glossary

- Agenome: agent genome containing prompt, persona/value weights, tool permissions, decomposition policy, and spawn budget.
- Candidate idea: shared unit of work generated and improved by the organism.
- Cross-domain transfer: candidate idea mapping a technique or result from source domain A to target problem B.
- Zeitgeist synthesis: candidate idea proposing a thesis, product, or framing fitted to current signals.
- Energy: finite compute/tool/spawn allowance.
- Critic council: adversarial verifier panel.
- Fitness: structured score used to apply selection pressure.
- Fusion: reproduction from two parent agenomes and/or synthesized parent outputs.
- Replay: reconstruction of a run from persisted events without fresh model calls.

### Ambiguous Terms

[open question] "Held-out idea-quality rubric" needs concrete scoring dimensions and ownership.

[open question] "Objective check per domain" needs exact examples for both supported idea subtypes.

[open question] "Current signals" for zeitgeist synthesis needs a bounded data source definition.

[open question] The exact embedding/model/provider used for simple novelty scoring is not yet locked.
