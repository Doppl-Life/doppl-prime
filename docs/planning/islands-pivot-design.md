# Design: The Islands Pivot тАФ from a 1:1 engine to a case-study graph

> Generated by `/office-hours` on 2026-06-27 ┬╖ Status: **APPROVED** (operator, 2026-06-27) ┬╖ Mode: engineering design
> Grounded in the `doppl-pivot-recon` workflow (7 agents) + `docs/prds/06-case-study-intake-prototype-prd.md`.
> Related: `docs/planning/coevolution-climb-plan.md` (the judge-climb plan this reorders), `docs/planning/phase-j-v4-decision-package.md`.

**Architecture sentence stays true:** *it's not the agent тАФ it's the kernel that breeds the agents; the event log is the truth, and the held-out judge is the floor the organism cannot lift.* This pivot adds a layer ABOVE the run (the case study) and widens the run's TAIL (one winner тЖТ many), without moving the truth (append-only log) or the floor (immutable judge).

## 0. The decision

**Target = the full Islands Platform (option C).** Build it as three increments so the contract bottleneck and the live-embedding engine don't land in one risky push:

- **Increment 0 тАФ flip Phase J** (prerequisite, separate solo commit). Already decided (operator-delegated 2026-06-27): substantive bar, spread тЙе 0.50, v4 text accepted. The pivot's "judge picks top-N" step *consumes* judge acceptance scores, so a recalibrated, un-flattened judge is a prerequisite тАФ a flat ~0.53 judge makes top-N near-arbitrary.
- **Increment A тАФ Recoverable Islands MVP** (contract-free wedge).
- **Increment B тАФ Convergence Engine + durable case study** (Phase C folds in here).
- **Increment C тАФ Problem-recovery + bloom/prune/chat** (the stretch goals).

Each increment is independently valuable and B/C build directly on A. We commit to the destination (C); we do not commit to one monolithic build.

## 1. Problem statement (the pivot)

Today Doppl is **one-to-one and disassociated**: one problem string in тЖТ one run тЖТ evolve тЖТ **one** top-fitness output appended to the ledger and effectively orphaned (recoverable only by ID-spelunking). The pivot makes it a **graph / islands** system:

1. **Decouple identity.** The input becomes a reusable **case study** (scenario + maybe an implied problem, possibly raw conversation to derive a problem from), with its own durable identity. A **run** is one execution against a case study. Re-run the same case study тЖТ different run, same root тЖТ join on `case_study_id`.
2. **Many winners, not one.** Replace the single output with: cluster the final candidate pool by similarity тЖТ fuse each cluster тЖТ add one "disparity fusion" of the most-divergent ideas тЖТ **the judge picks the top N up to a configurable max** (default 2; max is a ceiling, so 0/1/2 allowed). Each winning **doppel** carries `case_study_id`.
3. **First-class graph.** Lineage becomes a queryable structure (case study тЖТ runs тЖТ doppels), recoverable by join, not orphaned rows.
4. **Stretch 1 тАФ split problem-recovery from solving.** A **goal switch** makes the same kernel emit N candidate **problems** from a case study; the human ratifies one; it's fed back as a new run tagged to the case study; the resulting doppel carries the trace `doppel тЖТ problem-recovery тЖТ case-study`.
5. **Stretch 2 тАФ bloom + chat.** Visualize case study тЖТ problems тЖТ solutions; human prunes bad nodes (append-only); chat over the survivors to tease out the through-line.

## 2. What recon established (and three corrections to the working model)

**Current behavior, grounded:**
- `run_id` is minted once at `apps/api/src/routes/runs.ts:179` (POST /runs) and is the *only* identity for an execution тАФ the per-run monotonic `sequence` partition key, the prefix of every derived id, the sole join key of every projection.
- The "problem" today is a single free-text string: `RunConfig.seed` (`packages/contracts/src/run/run-config.ts`), carried in the `run.configured` event payload. A boot catalog (`ProblemSet {id,title,prompt}`, `runtime/config/configSchema.ts`) exists but is ephemeral and its `id` is **discarded** at launch (`apps/web/src/components/run/runConfigForm.ts:buildRunConfig`).
- The append-only log (`run_events`) is authoritative and physically immutable (the `migrations/0001` triggers reject UPDATE/DELETE/TRUNCATE). Every projection is an in-memory pure fold rebuilt-on-read; the 11 declared derived tables are vestigial and unwritten.

**Correction 1 тАФ there is no end-of-run fusion today.** Fusion (`selection/reproduction/fuse.ts`) is a *per-generation* breeding operator over **agenomes**, skipped on the final generation (the FINAL-GENERATION GUARD, `generationLoop.ts:981-1004`). The single output is just the **top-`total`-fitness scored survivor** (`terminalClassifier.ts` тЖТ `partialSummary.ts:bestScoredSurvivor`, tie-break lowest sequence), recorded as the scalar `run.completed.finalIdeaRef`; the `'selected'` status is *derived* post-hoc by `projections/reducers/winner.ts:winnerReducer`. тЗТ The cluster-then-fuse step is **net-new** (nothing to remove), and a thin "top-N winners, no fusion" version is legitimate.

**Correction 2 тАФ the multi-winner result shape costs zero contract bumps.** `run.completed` and `run.configured` are NOT in `HIGH_TRAFFIC_PAYLOAD_MAP` (`packages/contracts/src/events/payload-map.ts` тАФ 8 keys, neither present тЖТ both resolve to the generic `z.record` payload), so `case_study_id` and `finalIdeaRefs[]` ride the generic JSONB payload with **no `schemaVersion` bump** (LESSONS ┬з107). The winner is *not* its own event today: `'selected'` is a derived `CandidateStatus` that `winner.ts:winnerReducer` synthesizes from the scalar `run.completed.finalIdeaRef`. **There is NO `candidate.selected` event type** (the registry is 41 members; `candidate.selected` is not one тАФ the in-registry-but-unemitted precedent is `candidate.rejected`). тЗТ Increment A keeps winners on the generic `run.completed` payload as `finalIdeaRefs[]` and touches **zero frozen contracts**; minting a per-winner event is a registry-add deferred to Increment B's bump batch.

**Correction 3 тАФ clustering is degenerate without a live embedding key.** Recorded/keyless runs return a fixed 8-dim stub vector for every candidate (`model-gateway/stub/fixtures.ts`); discriminating 1536-dim vectors only exist on live runs with `OPENAI_API_KEY`. The distance machinery already exists (`selection/reproduction/parent-distance.ts:selectDistantPair` IS the disparity-fusion selector; `selection/novelty/cosine.ts` is pure/replay-safe); a clustering primitive is ~40-80 lines. тЗТ The cluster-fuse story only demos for real on a **live** run; lexical-Jaccard is the keyless degrade.

**The one deep unknown:** the log partitions strictly by `run_id`, but a **case study spans many runs** тАФ it has no natural home in the log. Increment A dodges this by stamping `case_study_id` into the run's own config payload and reconstructing the graph by join (the first-ever cross-run projection, composing per-run folds тАФ `buildProjection` throws on `mixed_run` by design). "Where authoritative case-study state lives" is the real B-phase decision (see ┬з7).

**De-riskers:** PRD-06 already specifies the case-study packet (id/title/prompt/context/constraints/evaluator-anchor + agent-visible vs evaluator-only boundary). React Flow already renders multiple `selected` nodes (`apps/web/.../lineageToFlow.ts`), so the UI is not single-output-shaped.

## 3. Premises (agreed in office hours, 2026-06-27)

1. The recoverable graph is worth more than the fusion engine, first тАФ a traversable case-study тЖТ runs тЖТ doppels graph is independently valuable even when winners are top-N-by-fitness.
2. **"The judge picks top-N" тАФ split by increment.** (Load-bearing safety call; see ┬з4.)
   - **2a (Increment A):** pure re-sort of the per-candidate acceptance scores the judge ALREADY produced тАФ NO new judge call. The judge runs over the whole final-generation pool before the FINAL-GENERATION GUARD, persisting one `JudgeResult.acceptance` scalar per candidate, read verbatim by selection. Top-N is a deterministic `.filter`/sort joined by `candidateId`.
   - **2b (Increment B):** the cluster-fuse stage produces NET-NEW fused candidates that did not exist when the judge ran, so they DO get new judge invocations тАФ under the same byte-identical rubric, runner-computed acceptance (rule-#6-clean), but genuinely new calls, not a re-sort.
3. The cluster-fuse demo runs live (keyless = lexical degrade).
4. Phase J flips before any top-N selection.
5. Stretch-1 problem-recovery uses **human ratification**, not a second immutable rubric.

## 4. Safety design decisions (load-bearing тАФ by rule)

- **Rule #6 (immutable judge) тАФ the judge stays a *consumer*, never a new authority, in BOTH increments.** In **A**, multi-winner "top-N" RE-SORTS the existing per-candidate `judge.reviewed` acceptance scores (no new call). In **B**, the cluster-fusion produces net-new fused candidates that get scored by **new** judge calls under the **same byte-identical rubric** (runner-computed acceptance, no new axis, no ranking prompt) тАФ new invocations, not a new authority. In neither case does an agent or config gain a lever over the judge. `ScoringPolicy`/`FinalJudgeRubric`/`FinalJudgeAxis`/`JudgeResult` stay byte-identical. The clustering + cluster-fusion + disparity-fusion run as PRE-judge candidate *preparation*; the top-N cut is a deterministic cap on judge output.
- **Rule #1 (caps kernel-enforced) тАФ MAX-N and the 0-winner floor are caps, not knobs.** The default-2 / max-N ceiling and the acceptance floor below which zero winners are crowned are kernel/config-enforced like population/energy caps тАФ never prompt- or judge-settable. The net-new end-of-run fusion stage *generates* candidates (debits energy + spawns agenomes), so it must clamp to remaining headroom via the existing `selection/allocation.ts:allocate` spawn-clamp, and it must be placed carefully because it REVERSES the FINAL-GENERATION GUARD.
- **Rule #2 (append-only) тАФ every new concept is an event, pruning is never a delete.** Case studies, problems, N winners, and pruning are all new events folded into derived projections. "Prune a bad node" = a `node.pruned` append-only event folded to a hidden status (the `lineage.culled тЖТ 'culled'` precedent), with the projection filtering it out. The cross-run graph is derived/rebuildable, never stored authoritatively.
- **Rule #5 (candidate text is data) тАФ the withheld-solution boundary (PRD-06).** A case study's evaluator-only / withheld-solution fields must never leak into generation prompts. The case-study scenario reaches generation only as `wrapUntrusted` DATA (the seed already does, per the existing isolation). Raw-conversation-to-problem derivation keeps the conversation as untrusted data.
- **Rule #7 (replay calls no providers) тАФ clustering must be replay-deterministic.** Clustering over already-persisted vectors is pure/free. Any stochastic step must be eliminated (deterministic single-linkage / index-canonical iteration, like `selectDistantPair`) or persist its RNG outcome (the `createRng(seed) + persistOutcomes` bridge). The judge top-N reads only persisted numeric acceptance scores; the fused candidates persist their synthesis outcomes.

## 5. The increments

### Increment 0 тАФ Flip Phase J (prerequisite, separate solo commit)
Move `JUDGE_AXIS_CRITERIA_V4` (`apps/api/test/eval/criteria-v4.ts`) into `src/verifier/judge/judge-core.ts` (replace `JUDGE_AXIS_CRITERIA`), bump `rubric.ts` `policyVersion` mvp-3 тЖТ `final-judge-v4`, re-record the 6 mvp-3 fixtures, keep the frozen contract-immutability tests green UNEDITED. One confirming live `pnpm test:eval` first (the good tier was fixed keyless but not re-measured live; target: good ~0.64 between mediocre and excellent). Zero pivot dependency.

### Increment A тАФ Recoverable Islands MVP (contract-free)
**Goal:** a traversable case-study тЖТ runs тЖТ doppels graph + N-best winners, touching zero frozen contracts.

1. **Carry `case_study_id`** тАФ stop discarding `ProblemSet.id` in `apps/web/src/components/run/runConfigForm.ts:buildRunConfig`; thread it through `POST /runs` (`routes/runs.ts`) onto the generic `run.configured` JSONB payload (LESSONS ┬з107, no bump). `projections/run-summary.ts:buildRunSummary` reads `payload.case_study_id` back alongside `seed`. Re-run = new `run_id`, same `case_study_id`. A `GET /case-studies` list evolves from the **existing** `routes/problem-sets.ts` (`GET /problem-sets`) to surface re-runnable case studies.
2. **N-best winners (no fusion yet)** тАФ `terminalClassifier.ts` picks the top-N scored survivors by **judge acceptance** (post-Phase-J), bounded by a kernel `maxWinners` cap (default 2) and an acceptance floor (0 allowed). Persist as **`finalIdeaRefs[]` on the generic `run.completed` payload** (no bump тАФ this is the path that keeps A contract-free; a per-winner event is deferred to B). Touched code: `reducers/winner.ts` тАФ `finalIdeaRef()` parses an array and `winnerReducer` loops to mark the SET `'selected'` (a real change, not free тАФ today it reads one scalar); `apps/web/.../finalIdeaData.ts:selectWinner тЖТ selectWinners` (`.find тЖТ .filter`); React Flow renders N `selectedWinner` nodes unchanged (`lineageToFlow.ts` is a per-node map).
   - **0-winner terminal shape (specified):** if scored survivors EXIST but none clear the acceptance floor, the run still `run.completed` with **`finalIdeaRefs: []`** (a valid "island with no crowned doppel" тАФ meaningful for the graph). This is DISTINCT from `run.failed{no_scored_survivor}`, which stays the case where there are no scored survivors at all. The acceptance floor is itself a rule-#1 kernel cap (┬з4), never judge/prompt-settable.
3. **First cross-run projection** тАФ a NEW `apps/api`-internal projection `projections/case-study-graph.ts` (mirroring `RunSummaryItem` / the research-knowledge-graph shape тАФ NO frozen contract) composing N per-run current-state folds (one `readByRun` per run, stitched тАФ never a mixed-run fold, since `buildProjection` throws on `mixed_run`), exposed via a new route in `routes/runs-read.ts` at `GET /case-studies/:id/graph` (fed by a case-study-grouped extension of `run-list.ts:listRunIds`). A bloom graph view in `apps/web` (React Flow, the proven 2nd-graph-view pattern, cf. `KnowledgeGraph.tsx`).

**Result:** the islands graph is recoverable and visual; winners are the N-best (not yet clustered/fused). Append-only-clean, zero frozen-contract changes.

### Increment B тАФ Convergence Engine + durable case study (Phase C folds in)
**Goal:** make the N winners *diverse and synthesized* (the real "convergence"), and promote the case study to first-class.

1. **The net-new end-of-run cluster-fuse stage** between the loop's last generation and `classifyRunTerminal` (`generationLoop.ts:1102`): cluster the final scored pool over the existing 1тИТcosine matrix (a new ~40-80-line pure clustering primitive beside `parent-distance.ts`) тЖТ fuse each cluster via `fuse.ts` тЖТ one disparity-fusion via `selectDistantPair` тЖТ the judge scores the fused candidates тЖТ top-N by acceptance. **Cap-aware** (reverses the FINAL-GENERATION GUARD; energy/population clamp), **replay-deterministic** (no unpersisted RNG; persist synthesis + any clustering outcome). **Demo live** for real cluster discrimination; lexical-Jaccard degrade keyless.
2. **Phase C merges in here.** Phase C's "divergeтЖТconverge anneal" and this terminal cluster-fuse are two halves of one convergence story: explore divergently across generations (Phase C's per-generation `successor.ts` pressure, coupled to judge-score variance тАФ now meaningful post-v4), then converge/cluster/fuse at termination (this stage). Build them together; do not ship Phase C as a standalone phase on the old engine.
3. **Durable case study** тАФ promote from the ┬з107 payload-join to a first-class `case_study.created` event (RunEventType registry-add тЖТ the FIRST `schemaVersion` bump) with its own authoritative home. Resolves the run_id-partition tension (see ┬з7).

### Increment C тАФ Problem-recovery + bloom/prune/chat (the stretches)
1. **Goal switch (Stretch 1)** тАФ optional `goalMode: 'solve' | 'recover_problem'` on `RunConfig` (default `solve` = byte-identical today; one bump). At `generationLoop.ts:buildPopulationRequest`, the `schema:` arg + system framing are already per-call params, so `goalMode` swaps `CandidateContent` for a new **`CandidateProblem`** model + a problem-eliciting framing. The kernel emits N candidate problems; the **human ratifies** one (a new ratification command/event тАФ NO automated judge, per premise 5); the ratified problem becomes a new run's seed referencing the same `case_study_id` and a `problem_recovery_id`. The doppelтЖТproblemтЖТcase-study trace is a 2-hop join.
2. **Bloom + prune + chat (Stretch 2)** тАФ bloom React Flow over the cross-run projection; prune = a `node.pruned` append-only event + projection filter (NEVER a delete); chat-over-corpus reuses `retrieveNotes` over the survivor corpus (lexical today; cosine free if note vectors get persisted тАФ `retrieve.ts:83-90` auto-upgrades). Survivor text stays `wrapUntrusted` DATA in the chat call (rule #5).

## 7. The deep open decision тАФ where case-study authoritative state lives (B-phase)
A case study spans many runs; the log partitions by `run_id`. Three options:
- **(a) New append-only stream/table parallel to `run_events`** with its own per-case-study `sequence`. Cleanest separation; most new infrastructure.
- **(b) Synthetic `case_study` run_id partition** for `case_study.*` events тАФ reuses `run_events` + the existing append/sequence machinery; bends the "run_id = a run" meaning.
- **(c) Derived projection over `case_study.*` events** that ride a designated partition тАФ lightest, but needs a home for those events.
**Recommendation:** start A with the payload-join (defers this entirely); at B, lean toward **(a)** for a clean first-class entity, falling back to **(b)** if reusing the append machinery proves materially cheaper. This is the single biggest architectural call and affects every downstream projection тАФ escalation-worthy at B.
