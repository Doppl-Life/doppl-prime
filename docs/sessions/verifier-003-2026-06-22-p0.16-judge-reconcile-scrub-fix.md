### verifier-011 — the scrub fix (frozen P0.2 amendment, user Option-B)

**Bug (confirmed in `packages/contracts/src/security/redaction.ts`):** `SENSITIVE_KEY_FRAGMENTS`
contains `'token'`, matched case-insensitive `includes`, and line 88 whole-redacted a sensitive
key's value **regardless of type**. `ProviderMeta.tokensIn`/`tokensOut` (`z.int().nonnegative()`)
both contain "token" → redacted to the **string** `'[REDACTED]'` → the persisted payload fails
its frozen contract's `safeParse` on read. Because the payload-map narrows the **pre-scrub**
payload, the append SUCCEEDS and the corruption is **silent** — surfaced only on read/replay
(breaks rules #2/#4/#7). verifier-010's `judge.reviewed`←`JudgeResult` was the first event to
persist a `ProviderMeta` in a payload; kernel's `energy.spent`←`EnergyEvent` (`providerMeta?`)
will hit the identical corruption at P3.

**Fix:** redact a sensitive-key value **unless it's a `number`/`boolean`** (never a credential) —
`isSensitiveKey(key) && typeof child !== 'number' && typeof child !== 'boolean' ? REDACTION_PLACEHOLDER : scrubValue(child)`.
This preserves whole-redaction for strings AND objects/arrays (no weakening — the naive
`typeof === 'string'` variant would recurse object-valued secret keys and LEAK a non-pattern
secret), while letting numeric metadata through. Logic-only; **no schemaVersion change**; the P0.2
redaction snapshot was re-recorded. Built as a SOLO safety-invariant slice (rule #4) with a
`security-reviewer` pass.

### verifier-010 — P4.8 judge reconcile to P0.16

  kernel P1/P2 + demo + P0.16); merging it would entangle the bounded reconcile with a full-trunk
  integration. `e664f68` is contracts-only, so a cherry-pick pulled JUST the P0.16 contract.

  seals fully green now) over routing to a re-activated contract track; the fix propagates to all
  tracks via the verifier→cody merge.
