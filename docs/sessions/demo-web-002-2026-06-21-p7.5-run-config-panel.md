## Decisions made
- **Cap-max = fail-closed DEFENSE/UX, lowering-only** ‚Äî `clampCap` clamps to `[1, CAP_CEILING]`; `validateForm` rejects any cap above the ceiling and never submits it. `CAP_CEILING` is a client constant mirroring the API `defaultConfig.caps`; the **API (P6.6) + kernel are the authoritative enforcers** (rule #1). Validate SHAPE via the frozen `RunConfig` Zod ‚Äî never re-implement the contract.
- **Idempotent Start** ‚Äî the panel generates a per-submit `crypto.randomUUID` key passed to `runClient.startRun` ‚Üí `Idempotency-Key` header; the API dedups (not re-implemented). The button disables while-starting + after-success so a double-click can't fire a second `startRun`.
- **validate-on-submit (vs the prototype's hard-disable)** ‚Äî Start is clickable; an invalid submit surfaces inline accessible errors that say WHY (the prototype's disabled button can't). Cap-over-ceiling + all-subtypes-off are still prevented at the input/toggle. (Recorded by the orchestrator as a prototype-fidelity improvement.)
- **wall-clock UX in minutes** ‚Üí converted to the contract's `wallClockTimeoutMs` (√ó60 000) at build time.
