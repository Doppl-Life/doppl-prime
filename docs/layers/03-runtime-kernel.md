# Runtime Kernel

## Executive summary

This is the "weird part" of Doppl — a custom TypeScript kernel that is deliberately **not** bent into an off-the-shelf workflow framework. It is the brain that breeds the agents: it drives every run through four lifecycle state machines (run, generation, candidate, agenome), enforces the hard caps and the kill switch that keep a self-modifying multi-agent loop finite, runs the energy "metabolism" that only debits successful work, owns the single seeded random-number generator that makes a whole run reproducible, and contains the in-process worker that actually executes a run and recovers cleanly after a crash. Crucially, the kernel is the **sole emitter of authoritative lifecycle events** — when a generation starts, an agenome spawns, or a run finishes, those facts are written by the kernel and nobody else. The subsystems that critique, score, and reproduce ideas plug in as injected "seams"; the kernel orchestrates them but treats their events as data it reads back, never as something it authors. Almost every module here is a **pure decision/computation** (no I/O, no clock, no randomness) — the generation loop and the worker are the only places that emit events and call providers, which is what makes the rest replayable and testable.

Two newer capabilities now ride **inside the loop** without breaking that posture. (1) The kernel holds an injected, **default-absent in-run knowledge-retrieval port** — before each agenome generates, it can read the run's accumulated research notes (the "shared knowledge space") and thread the snippets into that one generation request as untrusted data; absent the port it is byte-identical to the old behaviour. (2) The kernel hosts the **ratchet + mutagen-strategy** scaffolding for the "climb" (a reigning-champion ledger, a champion-as-extra-parent decision, and the mutation-strategy params the loop reads) — but the directed-reproduction / convergence / cull mechanics that *use* those signals live in selection (`05`). Generation itself is now run **concurrently** across an agenome batch, bounded by an energy-headroom-clamped ceiling, with the per-generation kill staying the authoritative finiteness enforcer.

## Responsibilities

- **Owns the four lifecycle state machines** (run / generation / candidate / agenome) as pure `(from,to)→allowed` transition guards — the table *is* the §3 spec (`apps/api/src/runtime/state/transitionGuard.ts:37`).
- **Enforces caps in the kernel, never in prompts** — population, generations, energy, spawn-depth, tool-calls, wall-clock + the kill switch (`apps/api/src/runtime/caps/`), and clamps the `spawnBudget` allocation hint (`apps/api/src/runtime/spawn/spawnBudgetClamp.ts`). Tool-calls are also backstopped **inline in the loop** (the relay gate, see below).
- **Runs the energy ledger** — success-only `doppl_energy` accounting, estimate-then-reconcile (`apps/api/src/runtime/energy/`).
- **Owns the single seeded PRNG + the outcome-persistence bridge** for deterministic replay (`apps/api/src/runtime/rng/`).
- **Drives the generation loop** — the bounded happy-path + failure/kill control flow, the SOLE emitter of authoritative lifecycle events, now with **bounded concurrent agenome generation** (`apps/api/src/runtime/loop/generationLoop.ts`).
- **Holds the in-run knowledge-retrieval PORT** (`RetrieveKnowledge`, default-absent) the loop calls per-agenome before generation, persisting the retrieved note-id set and threading snippets into the population request — the *feature* (projection + kNN + boot bridge) is documented in `11`; the kernel only owns the seam shape + the rule-#7 carrier.
- **Hosts the ratchet/climb scaffolding** in `runtime/loop` — the reigning-champion ledger (`championLedger.ts`), the champion-as-extra-parent decision (`withChampionParent`), and the mutagen-strategy params (`mutagenStrategy.ts`). The selection-side directed-repair / convergence / truncation-cull / elitism mechanics that consume these are `05`'s.
- **Classifies the run terminal** and **recovers from crashes** at boot (`apps/api/src/runtime/terminal/`, `apps/api/src/runtime/recovery/crashForward.ts`).
- **Hosts the in-process single-active-run worker** + idempotency + the boot composition root (`apps/api/src/runtime/worker/`, `apps/api/src/boot/`).
- **Is NOT:** the critic council, the held-out judge, the scoring/novelty math, or fusion/mutation/cull/elitism/directed-repair — those are the verifier (`04`) and selection (`05`) layers, wired in as seams. The kernel **never critiques, checks, or scores a candidate itself** (`generationLoop.ts:142`). It also is **not** the append writer (event-store, `01`) nor the provider SDK boundary (model-gateway, `02`) nor the KB substrate itself (`11`) — it depends on all of them through ports.

## Interfaces & contracts

**Inward dependencies (what the kernel consumes):** the frozen Zod contracts from `packages/contracts` are the kernel's type vocabulary — `RunStatus`, `GenerationStatus`, `CandidateStatus`, `AgenomeStatus`, `RunCaps`, `RunConfig`, `Agenome`, `CandidateIdea`, `EnergyEvent`, `ProviderMeta`, `RunEventType`, `GenerationOperator`, `CURRENT_SCHEMA_VERSION`, `wrapUntrusted`. TS types derive from the schemas (`z.infer`), never the reverse.

**The event-store port (`01`):** the kernel reaches Postgres only through the `EventStore` interface — `append(input)` and `readByRun(runId)` (`apps/api/src/event-store/append.ts:49`). It never imports a provider SDK and never writes `run_events` outside this port. `RunEventRow` (`append.ts:47`) is the persisted row shape the kernel folds over.

**The model-gateway port (`02`):** the loop sees a runtime-local `GenerationGateway` (`generationLoop.ts:189`) that *composes* the frozen `ModelGateway` — it carries the `ModelGatewayResponse` plus observability channels (`ToolCallObservation`, attempt failures) the frozen response cannot represent. `generate(request, {toolBudget?})` takes the kernel-clamped tool-call budget as a HINT (`GenerateOptions`, `generationLoop.ts:181`); a tool-orchestrating gateway self-limits to it, a pass-through gateway ignores it (see Safety, rule #1).

**The subsystem seams (verifier `04` / selection `05`):** the loop delegates verify/score/reproduce to three injected ports (`GenerationSeams`, `generationLoop.ts:228`):

```text
VerifySeam    = (candidates, ctx)            => Promise<void>
ScoreSeam     = (candidates, ctx)            => Promise<void>
ReproduceSeam = (ctx{parents,outcomes,spawnBudget,mode}) => Promise<void>
```

Each seam **appends its own events** (`critic.reviewed`, `fitness.scored`, `lineage.culled`, `agenome.fused`/`mutated`/`reproduced`); the loop reads them back via `readByRun` and **never authors them** (`generationLoop.ts:921`,`926`,`1003`). This is the §2.5 subsystem boundary expressed as code shape (LESSONS §64).

**The in-run retrieval port (NEW, `11` owns the feature):** an additive, default-absent injected seam (`generationLoop.ts:280-309`):

```text
RetrieveKnowledge = (args:{runId,generationId,agenome}) => Promise<RetrievedKnowledge|undefined> | RetrievedKnowledge | undefined
RetrievedKnowledge = { noteIds[], snippets[], direction:'near'|'far', method:string }
```

The loop sees only this port (clean layering — the pure loop may not import `projections`/`selection`). When it returns ≥1 note, the loop persists the note-id SET on `candidate.generation_started` (the rule-#7 carrier) and passes the snippets to `buildPopulationRequest`'s `retrievedNotes` param; absent the port (or an empty result) the request is byte-identical to the baseline.

**Events the kernel emits (authoritative):** generation lifecycle (`generation.started/verifying/scoring/reproducing/completed`, `generation_failed`, `generation.skipped`), `agenome.spawned`, `agenome.failed`, `candidate.created`, `candidate.generation_started` (an op-start marker — now ALSO the rule-#7 carrier for the retrieved-note-id set when in-run retrieval is active), `energy.spent`, `provider_call_failed`, `llm_call_telemetry`, `tool_call.started/finished`, and the run terminals (`run.started/completed/failed/stopped/cancelled`, `energy_exhausted`). All carry `actor: 'runtime'` and `schemaVersion: CURRENT_SCHEMA_VERSION` (`CURRENT_SCHEMA_VERSION` is now **10**, the additive tool-use TU.1 bump; see `00`).

## Data & state

- **No state lives in the kernel.** The event log (`run_events`, Postgres) is the only durable state (safety rule #2). Every kernel decision is recomputed from the persisted log on read — `cumulativeSpend` folds `energy.spent` actuals, `resolveEligibleParents` folds `fitness.scored ∧ ¬lineage.culled` (`generationLoop.ts:378`), `reigningChampion` folds the cross-gen peak survivor, `classifyRunTerminal` reads the whole log.
- **`AppConfig`** (`apps/api/src/runtime/config/configSchema.ts:104`) — the one deep-frozen, all-`readonly` config handle: `runConfig`, `registry`, `scoringPolicy`, `caps`, `costMap`, `problemSets`, `seedSet`, **plus three new climb knobs** `mutationStrategy`, `eliteCount`, `hallOfFameCarry` (`configSchema.ts:116`). Carries **no credential field** (rule #4). Built once at boot by `loadConfig`.
- **`DEFAULT_CAPS`** (`configSchema.ts:39`) = `{maxPopulation:12, maxGenerations:6, energyBudget:12_000, maxSpawnDepth:4, maxToolCalls:600, wallClockTimeoutMs:1_200_000}` — the **boot ceiling**; per-run caps may only lower within it. **Resized for tool-use (B1 sizing)** from the pre-tool defaults: `energyBudget 1000→12000`, `maxToolCalls 64→600`, `wallClockTimeoutMs 600_000→1_200_000` (10 min → 20 min). `maxPopulation`/`maxGenerations`/`maxSpawnDepth` unchanged. The kernel still ENFORCES them — this only moves the *default* ceiling.
- **Env knobs (two projection paths).** The closed `ENV_ALLOWLIST` (`apps/api/src/runtime/config/envSchema.ts:38`) gained `DOPPL_MAX_TOOL_CALLS → caps.maxToolCalls` and `DOPPL_WALL_CLOCK_MS → caps.wallClockTimeoutMs` (`envSchema.ts:44-45`); these are documented in `.env.example`. The three **climb knobs** are read by a SECOND path — `loadConfig` reads `env.DOPPL_MUTATION_STRATEGY`/`DOPPL_ELITE_COUNT`/`DOPPL_HALL_OF_FAME_CARRY` **directly** (`loadConfig.ts:126/131/136`), bypassing the allowlist and the `.env.example` drift guard (see Gotchas). Parsers (`loadConfig.ts:155`/`:164`) clamp to `[0, maxPopulation]`; absent/garbage → `mutationStrategy=fusion_only`, `eliteCount=1` (**elitism ON by default**), `hallOfFameCarry=0` (**ratchet OFF by default = HEAD-identical**).
- **`DEFAULT_COST_MAP`** (`apps/api/src/runtime/energy/costMap.ts:24`) = `{tokensPerUnit:1000, perToolCall:5, perSpawn:50}` — the §4 energy cost map.
- **`DEFAULT_SEED_SET`** (`apps/api/src/runtime/seed/seedAgenomes.config.ts:35`) — four authored gen-0 personas (analogist, first-principles skeptic, zeitgeist synthesist, pragmatic builder); `SeedAgenomeTemplate` carries only the 5 trait fields, single-sourced from the frozen `Agenome` shape. Each persona now also carries a dominant `lens.<operator>` entry (≥0.5) in its `personaWeights` (`seedAgenomes.config.ts:31`), **active only under the `mutate_lens`/`adaptive` strategies** — inert (byte-identical) under the default `fusion_only`.
- **The outcome log** (`persistOutcomes.ts:45`) — an ordered, JSON-safe list of `{label, value}` RNG outcomes that the reproduce seam embeds into `agenome.fused`/`mutated` payloads so replay reconstructs draws without a PRNG.
- **The champion stash** — `championAgenome` (`generationLoop.ts:610`) is in-loop mutable state: the reigning champion's real `Agenome`, captured in the generation it is crowned. It is re-derivable on replay from the deterministic population + log (rule #7), and is only ever *read* — used by `withChampionParent` when the ratchet is on.
- **In-memory side state (non-authoritative):** the worker heartbeat (`heartbeat.ts`) and the operator-stop latch (`apps/api/src/boot/operatorStop.ts`) are side signals, never `run_events`. `pLimit`'s active-count + FIFO queue (`concurrency/pLimit.ts:26-27`) are purely in-process scheduling state.

## Dependencies

- **Depends on:** `packages/contracts` (`00`, frozen Zod schemas / status enums — the kernel's whole type vocabulary), the **event-store port** (`01`, `append`/`readByRun` — the only path to durable truth), the **model-gateway port** (`02`, structured-output validate/repair/reject for generation), and the new **`concurrency/` shared util** (`pLimit`/`mapLimit` — a bottom-leaf pure scheduler, importable from anywhere). It depends on these by *interface* — domain/runtime code never imports a provider SDK (rule #9, `apps/api/CLAUDE.md` forbidden-pattern #2).
- **Used by:** the **verifier** (`04`) and **selection** (`05`) subsystems are wired *into* the loop as seams at the boot composition root (`composeRuntime.ts:179`/`:192`/`:201`/`:215`); the **REST/SSE routes** (`07`) trigger `createStartRun` and poll the operator-stop latch; **projections** (`06`) fold the lifecycle events the loop emits; the **shared knowledge space** (`11`) wires its retriever into the loop's `RetrieveKnowledge` port (`composeRuntime.ts:226`); the **tool-use subsystem** (`12`) provides the optional tool-orchestrating generation gateway (`composeRuntime.ts:237`). The layer-dependency direction is strict: `routes → projections → selection/verifier → runtime → {event-store, model-gateway} → contracts`, with `concurrency/` a cross-cutting leaf (`apps/api/CLAUDE.md`).

## How it works (flow)

A run executes inside `runWorker → runGenerationLoop`:

```text
boot:  loadConfig (fail-fast) → migrate → crashForward → worker accepts work
                                              │
runWorker (worker/runWorker.ts:88):           │
  1. activeRunGuard      single-active-run, over the log     (:100)
  2. idempotency         start ONLY configured ∧ ¬started    (:110)
  3. append run.started  configured→running, guard-checked   (:121)
  4. heartbeat.beat()    side signal                          (:142)
  5. runGenerationLoop ──────────────────────────────────────┐
                                                              ▼
runGenerationLoop (loop/generationLoop.ts:436):
  materializeGen0 → spawn each agenome (cap-gated) + debit spawn energy   (:581,:588)
  for g in maxGenerations (cap-bounded loop):                              (:613)
    onIteration() heartbeat → detectKill (operator/energy/tool/wall-clock)(:615)
    generation.started                                                    (:630)
    ┌ per-agenome batch — CONCURRENT via mapLimit at an energy-clamped
    │  ceiling (:854,:855):
    │   processAgenome (:656): re-check detectKill → (optional) KB
    │     retrieve → persist candidate.generation_started{retrievedNoteIds}(:684,:690)
    │   → buildPopulationRequest (problem + KB notes as wrapUntrusted)     (:701)
    │   → gateway.generate(req,{toolBudget:max(0,maxToolCalls−consumed)})  (:712)
    │     accept ⇒ candidate.created + llm_telemetry + debit llm
    │     reject ⇒ provider_call_failed + agenome.failed (no debit)
    │     per surfaced tool call: inline enforceCap('maxToolCalls') gate   (:722)
    └      then relay tool_call.started/finished + debit tool iff ok≠false
    survivors<min ⇒ generation_failed                                     (:898)
    running[→degraded]→verifying → seams.verify                          (:917,:921)
    generation.scoring → seams.score                                      (:925,:926)
    resolveEligibleParents (fitness.scored ∧ ¬culled, agenome-keyed)      (:931)
    reigningChampion → stash championAgenome (if in this pop)             (:937,:946)
    0 survivors ⇒ generation.completed{survivors:0}                       (:955)
    compute spawnBudget = clamp(min(maxPop, energyHeadroom))              (:982)
    withChampionParent (hallOfFameCarry>0 adds champion as PARENT only)   (:991)
    generation.reproducing → seams.reproduce                             (:1000,:1003)
    backstop: count offspring > spawnBudget ⇒ cap_breach kill            (:1016)
    generation.completed [→ nextPopulation hook threads gen N+1,         (:1034,:1045)
                            clamped to maxPopulation]                     (:1056)
  classifyRunTerminal(log,killSummary?) → append ONE run terminal        (:1068)
```

Key paths to follow: a kill latched mid-generation is caught by the in-loop poll `maybeKillInLoop` (`generationLoop.ts:560`) so a runaway single generation cannot ignore a stop; under concurrency, `processAgenome` also re-checks `detectKill` and a latched `killTriggerInBatch` short-circuits queued agenomes (`generationLoop.ts:659`). At loop exit the terminal verdict is a no-op if a real terminal is already in the log (the kill path pre-emitted it via `executeKillAndDrain`, `loop/killDrain.ts:37`), so it never double-emits (`terminalClassifier.ts:126`).

## Design decisions & rationale

- **State machines as pure tables, one shared builder.** Each machine is a `Record<Status, Status[]>` table that *is* the ARCHITECTURE.md §3 spec, plus a terminal set; a full `Record` means a new enum member fails to compile until handled (`transitionGuard.ts:28`, LESSONS §33). Guards **decide only** — the loop emits, the appender persists (the §5 ownership split). Terminal-from is reported distinctly from an illegal pair so the kernel can treat a from-terminal attempt as a likely bug (`transitionGuard.ts:42`).
- **The kill switch is not a blanket "everything→failed."** §5's blanket phrasing does not map 1:1 onto the §3 tables, so `planKillSwitch` maps *each* non-terminal to its legal terminal and **excludes** already-terminalizing/transient states (`completing`, `stopping`, `degraded`) which drain through their own edge (`killSwitch.ts:62`, LESSONS §48).
- **Energy = success-only, by shape.** The frozen `EnergyEventType` has no failure member, so a failed attempt structurally cannot produce an `EnergyEvent` (`estimateReconcile.ts:6`, LESSONS §49). Finiteness rests on bounded retries + per-call timeout + wall-clock + count caps, *not* on energy debiting failures (ARCHITECTURE.md §4/§5).
- **One seeded PRNG, persisted outcomes.** Determinism is the demo's safety net. The kernel draws from a single per-run mulberry32 generator; the LIVE/REPLAY split is two *separate* factories so the replay path imports no PRNG and cannot re-sample even by accident (`persistOutcomes.ts:9`, LESSONS §47). The concrete algorithm IS the cross-machine byte-stability guarantee — a generator swap is **schemaVersion-relevant** (ARCHITECTURE.md §4:200).
- **The loop is pure orchestration over injected seams.** Selection (`05`) and verifier (`04`) emit their own events; the loop consumes them as data and never authors a seam-owned event (`generationLoop.ts:142`, LESSONS §64). This let the loop be built and tested against faked seams before the real subsystems merged. The same shape governs the new ports: the `RetrieveKnowledge` seam (`11`) and the tool-orchestrating gateway (`12`) are both injected — the loop sees only their port type.
- **In-run retrieval is an additive, default-absent seam.** The loop calls `deps.retrieveKnowledge` per agenome *only if wired*; absent it, no `candidate.generation_started` marker and no extra request message → byte-for-byte the old behaviour. When present, it persists the retrieved note-id SET (so replay re-threads identically — rule #7) and threads snippets as a SECOND `wrapUntrusted` user message (rule #5). The kernel deliberately holds *only* the port + the carrier; the projection, the kNN retriever, and the boot bridge live in `11`.
- **The ratchet, but as a parent-only nudge.** `reigningChampion` (`championLedger.ts:35`) is a pure cross-generation peak `scored ∧ ¬culled` candidate resolved verbatim from the log. `withChampionParent` (`generationLoop.ts:422`) re-presents its real agenome as an *eligible reproduction parent* when `hallOfFameCarry>0`, so the peak lineage cannot silently drop out and mean-revert — but it **never raises `spawnBudget`** (the offspring count is computed independently, `generationLoop.ts:982`), so the ratchet cannot inflate population (rule #1). Default `hallOfFameCarry=0` → the champion is never added (HEAD-identical).
- **Mutagen-strategy params live in `runtime/`, not `selection/`.** `mutagenStrategy.ts` (`strategyParams`/`agenomeLens`/`parseMutationStrategy`) is rng-free and read by *both* the boot config loader and the loop, and `runtime` may not import `selection` — so the params sit in runtime and selection imports them downward. The rng-using r/K decision, directed-repair, convergence controller, truncation cull, and elitism that *consume* these are selection's (`05`).
- **FB.3 mutagen operators / FB.4 diverge-converge dial** are generation-only framing controls. Operators map a closed enum → system-authored vetted fragments in the `population_generator` system message (`loop/generationOperators.ts:22`); the bias dial adds a band fragment + a clamped temperature (`loop/generationBias.ts`); the FB.4 bias *also* picks the in-run retrieval direction (near/far) at the boot bridge (`11`). All reach only the generation request — the critic/judge requests carry none (ARCHITECTURE.md §5:208–210).
- **Bounded concurrent generation.** Agenome generation — the only energy-debiting stage in a generation — now fans out via `mapLimit` at a ceiling clamped to the remaining-energy headroom (`floor(remaining/estPerCall)`, capped to population size and `DEFAULT_AGENOME_CONCURRENCY=6`, `generationLoop.ts:541`). A batch never STARTS more llm calls than the budget can pay for; per-call serialization is relaxed (see Gotchas: within-batch overshoot), with the generation-boundary kill as the authoritative rule-#1 enforcer.
- **Crash-forward, not resume.** MVP recovery forward-fails orphaned runs to a legal crash terminal rather than resuming (`crashForward.ts:9`, ARCHITECTURE.md §5:216); true idempotent resume is explicitly deferred.

## Safety & invariants

- **Safety rule #1 — caps are kernel-enforced, never prompt-enforced.** `enforceCap`/`enforceWallClock` take their ceilings *only* from the `RunCaps` argument and accept no trait/spawnBudget input, so nothing outside the configured caps can raise one by shape (`capEnforcer.ts:35`). `clampSpawnBudget` reduces the `spawnBudget` *hint* to `min(spawnBudget, max(0, remaining))` — it can only lower (`spawnBudgetClamp.ts:22`). The kill switch (`killSwitch.ts:111`) terminalizes on any breach. The reproduce backstop **counts** offspring the seam actually appended and triggers a `cap_breach` kill if a seam over-produced (`generationLoop.ts:1016`) — the un-bypassable enforcer. The champion-as-parent decision adds an eligible parent but **never raises `spawnBudget`**, and a threaded successor population is clamped to `maxPopulation` (`generationLoop.ts:1056`). **`maxToolCalls` is now kernel-enforced two ways:** the per-call `toolBudget=max(0,maxToolCalls−consumed)` passed to the gateway is a *clamped HINT*, and the authoritative backstop is the loop's inline `enforceCap('maxToolCalls')` relay gate (`generationLoop.ts:722`) — the (cap+1)th surfaced tool call is denied, not relayed/finished/debited, and latches a `cap_breach`; a `detectKill` fold over `tool_call.finished` (`generationLoop.ts:528`) is the boundary backstop. `mergePerRunConfig` re-clamps per-run caps to `min(posted, boot ceiling)` (`composeRuntime.ts:85`). The demo cap-override helper shares the exact `> maxima` boundary and refuses to ask for a raise (`demo/demo-cap-override.ts:14`, LESSONS §89).
- **Safety rule #8 — energy = successful productive spend only.** The energy ledger debits a productive call with a pre-call estimate then reconciles from the *real* `providerMeta` token counts (`reconcileEnergy`, `estimateReconcile.ts:68`); for llm a missing `providerMeta` throws rather than silently falling back to the estimate (`estimateReconcile.ts:75`). A failed/rejected call takes the `provider_call_failed` branch with **no** `energy.spent` append; a tool call debits energy only when its result is usable (`ok !== false`, `generationLoop.ts:766`) — a blocked/unavailable tool still counts toward `maxToolCalls` but debits nothing. In-run retrieval/embedding emits **no** energy event (it is not a productive spend); the energy markers and op-start markers debit nothing. `cumulativeSpend` folds ACTUAL, never estimate (`energyLedger.ts:53`).
- **Safety rule #7 — replay calls no providers and never re-samples.** The PRNG is the per-run mulberry32 seeded from `RunConfig.rngSeed` normalized `seed >>> 0` (`seededRng.ts:35`); `readRngSeed` pins that the seed comes from the persisted `run.configured` (`seededRng.ts:55`). `createReplayOutcomeSource` imports no PRNG and reconstructs each draw from the persisted outcome log, throwing loudly on label-mismatch/overdraw/out-of-range rather than re-sampling (`persistOutcomes.ts:93`). For in-run retrieval, the loop persists the retrieved-note-id SET on `candidate.generation_started`; replay re-folds that set (the `11` projection re-derives the `retrieved` edges) with **no** re-query — the boot retriever runs LIVE only. `classifyRunTerminal`, `reigningChampion`, `bestScoredSurvivor`, and `crashForward` are pure over the persisted log (no provider/clock/RNG in scope). The PRNG algorithm is treated as a contract: a swap silently breaks older replays (ARCHITECTURE.md §4:200).
- **Safety rule #2 — append-only, kernel is sole lifecycle emitter.** The loop appends only via the `EventStore.append` port (the `appendEvent` helper, `generationLoop.ts:445`); it emits only kernel-owned events and reads seam events back as data. Heartbeats and the operator-stop latch are deliberately *not* `run_events` (`heartbeat.ts:11`). Every transition (incl. the run terminal) is guard-validated through the §3.2 guards before append (`terminalClassifier.ts:171`, `runWorker.ts:112`). Concurrent appends from the batched generation still get a unique advisory-lock-serialized `sequence` (the per-run write lock is in the event-store, `01`); the loop's `eventSeq`/`energySeq` id-minting is collision-free because the read+increment is synchronous (no `await` between them).
- **Safety rule #5 — candidate input is isolated data.** The per-run problem reaches generation only as a `wrapUntrusted` user message; the trusted instruction (systemPrompt + `GENERATION_ISOLATION_FRAMING` + closed-enum operator/bias fragments) is the system message (`generationLoop.ts:53`/`:87`, LESSONS §91). **KB additions stay on the data side:** retrieved notes ride a SECOND `wrapUntrusted` user message and the only trusted addition is the candidate-independent `KB_RETRIEVAL_FRAMING` const (`generationLoop.ts:64`), which names the notes as DATA, never instructions; absent notes → no framing, no extra message (rule #5 / #6 — the snippets reach the `population_generator` request only, never the judge/critic isolation chokepoint).

## Gotchas & sharp edges

- **`energy_exhausted` is mid-flight, NOT a run-terminal.** An `energyBudget` breach drains in-flight work, scores the already-verified candidates, then the *real* terminal (completed if a survivor exists, else failed) follows — so the classifier's already-terminal no-op set excludes `energy_exhausted` or the run would be stranded non-terminal forever (`terminalClassifier.ts:52`,`:139`, LESSONS §69).
- **`degraded` (generation) and `repairing` (candidate) carry no event type.** They are state-machine-internal transient statuses; no `RunEventType` member carries them, so the event-log-derived current-state reducer adds no transition for them (ARCHITECTURE.md §3:167, LESSONS §62). The loop emits `generation.verifying` (with a `degraded:true` payload flag) after the `running→degraded→verifying` hop, not a `degraded` event (`generationLoop.ts:917`).
- **The reproduce seam is trusted to honor `spawnBudget`, but the kernel backstops it anyway.** The kernel computes the budget and passes it; an over-producing seam is detected post-reproduce and killed (`generationLoop.ts:1016`). Bug 6b714273 was a seam that hardcoded raw `maxPopulation` and minted a fresh full batch every generation — the count backstop is the fix.
- **Tool-IO can exceed `maxToolCalls` under concurrency, but the RECORDED ledger never does (confirmed, medium).** `maxToolCalls` caps the *recorded/debited* tool-call count (the authoritative `tool_call.*` events + energy debits). With up to 6 agenomes generating concurrently, each orchestrator reads a **stale** `toolBudget` snapshot of `max(0, maxToolCalls − toolCallsConsumed)` (`generationLoop.ts:712`) before any sibling increments the counter, so each self-limits to ~the full remaining budget and may execute real outbound web/x/youtube IO *before* the loop's relay gate sees it. The inline `enforceCap('maxToolCalls')` gate (`generationLoop.ts:722`) then only blocks RELAY/finish/debit and latches `cap_breach` — after the IO already happened. Net: outbound tool IO can over-run the cap (worst case ≈ ceiling × per-orchestrator budget), bounded loosely by wall-clock + per-call timeout + per-orchestrator `maxTurns`/budget. This is an **execution-vs-accounting gap, not a rule-#1 accounting bypass** — `toolCallsConsumed` increments synchronously so the persisted/debited count stays capped. (Full mechanism in `12`.)
- **Within-batch energy overshoot ~ceiling (confirmed).** Concurrent `processAgenome` tasks each re-check `detectKill` before generating, but several can pass the energy check then all debit, so a single generation's batch can exceed `energyBudget` by up to ~the concurrency ceiling before the next boundary kill fires (`generationLoop.ts:638-648`). The code accepts this — the ceiling is clamped to energy headroom so a batch never *starts* more than the budget can pay for, and the generation-boundary kill is the authoritative rule-#1 enforcer. Strict per-call cap serialization is relaxed vs the prior sequential loop.
- **DRIFT (confirmed, low sev) — a second env-projection path bypasses the closed allowlist + drift guard.** The three climb knobs (`DOPPL_MUTATION_STRATEGY`/`DOPPL_ELITE_COUNT`/`DOPPL_HALL_OF_FAME_CARRY`) are read **directly** off `env` in `loadConfig` (`loadConfig.ts:126/131/136`), NOT through the closed `ENV_ALLOWLIST` (`envSchema.ts`). They are therefore invisible to the `.env.example` drift guard (`env-example-drift.test.ts`, which checks only the allowlist ∪ credential ∪ boot-orchestration unions) and are **undocumented in `.env.example`** — yet all three are live-wired, not dead config. Benign (non-credential, int/enum-parsed, kernel-clamped to `[0,maxPopulation]`), but it weakens the single-closed-projection-path invariant (LESSONS §32/§95).
- **DRIFT (confirmed, low sev) — the `pLimit` header under-describes its callers.** `concurrency/pLimit.ts:7` leads with "Used to parallelize the ENERGY-FREE verifier + scorer fan-out", but `mapLimit` is also used for the energy-*debiting* agenome-generation fan-out (`generationLoop.ts:855`) and tool execution (`12`). The energy-free framing under-describes the actual callers; the energy-safety for the generation fan-out lives in the separate energy-headroom ceiling clamp (`generationLoop.ts:541`), not in `pLimit`. (`pLimit.ts:11` already acknowledges energy-debiting callers generally — only the lead sentence's caller list is incomplete.) `pLimit`/`mapLimit` propagate the first rejection like `Promise.all`.
- **In-run retrieval is LEXICAL-MVP + direction is asymmetric (feature in `11`).** The shipped retriever is Jaccard-lexical and keyless — no embedding call; the cosine path is dormant because `ResearchNote` persists no vector yet (pgvector not installed). `directionForBias` (`boot/knowledgeRetriever.ts:40`) is asymmetric: `far`/diverge fires ONLY when `generationBias > +0.2`; a strong converge lean (negative bias) AND neutral/absent both map to `near`. The retriever **self-gates** (returns `undefined` when the run has no notes), so wiring it unconditionally is byte-identical to the baseline on gen-0 / recorded / replay / any non-tool-using run.
- **DRIFT (harmless, doc-stale):** ARCHITECTURE.md §3:139/§5:206 phrase the kill as `{any non-terminal} → failed`; the shipped `planKillSwitch` does a per-state mapping (`configured→cancelled`, `running→stopping|failed`, `pending→skipped`) — the code is more precise and the per-state mapping is the intended behavior (killSwitch.ts header + LESSONS §48 document this divergence explicitly).

The count in those comments is stale; the behavior (heartbeat is not in the registry) is unchanged.
