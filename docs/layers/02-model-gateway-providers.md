# Model Gateway & Providers

## Executive summary

Every time the system needs an AI model — to generate an idea, critique one, embed text for novelty, judge a candidate, fuse two parents, or search the web for prior art — it goes through **one door**: the `ModelGateway`. The gateway is a thin, vendor-neutral seam. Domain code asks for a *role* ("I need a `population_generator`") plus an optional output schema, and the gateway figures out which provider and model to call, makes the call (with bounded retries and a timeout), and — critically — **validates the model's reply against the schema before handing anything back**. A reply that fails validation gets exactly one repair attempt; if it still fails, it is rejected. Nothing unvalidated ever flows onward toward the event log.

This layer exists to keep two promises. First, *the kernel never touches a vendor SDK* — OpenAI, OpenRouter, Ollama and friends live only inside small adapter files behind the port, so the rest of the codebase stays testable, replay-safe, and swappable. Second, *model output is untrusted until proven valid* — the gateway's validate / repair(≤1) / reject discipline is the authoritative gate, and candidate text is always carried as data, never as instructions. It is also the **single fan-out hub** in the system: many subsystems depend on it, it depends only on the frozen contracts. This doc describes the role→route registry, the OpenRouter / OpenAI-embedding / Ollama / retrieval adapters, the structured-output discipline, and the per-run model-route override that is allowlist-clamped so a run can never swap the held-out judge's model.

As of the **TU.1 amendment (schema v10)** the gateway also carries an **agent research-tool surface**: a `population_generator` request can offer a *closed allowlist* of research tools (`web_search` / `fetch_url` / `x_search` / `youtube_search`), and when the model asks to call one, the gateway surfaces that request *without* running the validate/repair/reject discipline (there is no final answer to validate yet). This doc owns the gateway/adapter **mechanics** of that surface — the frozen tool allowlist, the pure synchronous SSRF gate for `fetch_url`, the shared OpenAI-protocol message mapper, and the tool-call branch in the discipline. The **multi-turn orchestration** (driving the model↔tool loop, budgets, re-injecting results) and the **live research-tool implementations** (web/X/YouTube search, the pinned DNS-revalidating HTTP fetch) live one layer up in boot — see [12-tool-use-research.md](12-tool-use-research.md), which owns the full story.

## Responsibilities

- **Owns the provider seam.** Exposes the `ModelGateway` port (`call` + `capabilityFor`) whose I/O are exactly the frozen `ModelGatewayRequest`/`ModelGatewayResponse` contracts — no vendor type leaks into domain/runtime code (`apps/api/src/model-gateway/port.ts:19`).
- **Resolves role → route.** A registry maps each of the 7 `ModelRole`s to a `{provider, modelId, capability, fallbackRouteIds}` route, validated at boot (`apps/api/src/model-gateway/registry.ts:79`).
- **Authoritatively validates model output.** Runs the validate / repair(≤1) / reject discipline and returns a frozen `ModelGatewayResponse` (`apps/api/src/model-gateway/structured-output.ts:110`).
- **Confines vendor SDKs to adapters.** OpenRouter, direct-OpenAI embeddings, keyless Ollama, and a pluggable retrieval seam — each behind a client factory (`apps/api/src/model-gateway/adapters/*`).
- **Enforces finiteness per call.** Bounded retries + per-attempt timeout + one fallback, then a terminal `ProviderCallError` (`apps/api/src/model-gateway/adapters/retry.ts:96`).
- **Clamps per-run route overrides** to a frozen allowlist that excludes `final_judge` (`apps/api/src/model-gateway/model-route-override.ts:54`).
- **Offers the closed research-tool allowlist + surfaces tool-call requests (TU.1).** A frozen `TOOL_REGISTRY`/`TOOL_IMPLS` pair gates which tools exist (the rule-#3 analog of the check-runner allowlist); the OpenRouter adapter offers them on a `population_generator` request and allowlist-filters any returned `tool_calls`; the gateway surfaces a tool-call turn *without* validating (no output yet) (`apps/api/src/model-gateway/tools/registry.ts:236`, `gateway.ts:77`).

It is explicitly **NOT** responsible for:
- **Energy accounting / event emission.** The gateway debits no energy, emits no events, persists nothing — it returns `providerMeta` + `validationResult` and the kernel does the accounting (`structured-output.ts:14-16`). Safety rule #8.
- **Secret redaction.** It carries no credential field; the persistence boundary scrubs the raw `output` before append/Langfuse (`structured-output.ts:14`).
- **Replay.** Replay uses the recorded fake, never this live path (`live-gateway.ts:48`). Safety rule #7.
- **Choosing the judge's rubric or scoring policy.** Those are immutable to agents (safety rule #6) and live elsewhere.
- **Driving the multi-turn tool loop or executing tools.** The gateway *offers* tools and *surfaces* a tool-call request; the boot-layer tool-orchestrator drives the model↔tool loop (re-asking, budgeting), and the live tool implementations (web/X/YouTube search, the pinned HTTP fetch) live in boot/`toolSeams`. See [12-tool-use-research.md](12-tool-use-research.md).

## Interfaces & contracts

**The public port** (the only surface domain code sees):

```ts
interface ModelGateway {
  call(request: ModelGatewayRequest): Promise<ModelGatewayResponse>;   // never throws on provider failure
  capabilityFor(role: ModelRole): ProviderCapability;
}
```
(`apps/api/src/model-gateway/port.ts:19`)

**The frozen wire contracts** (all from `@doppl/contracts`, never redefined here):

- `ModelGatewayRequest` = strict `{ role, prompt XOR messages, schema?, maxTokens?, samplingParams?, tools? }` (now a 7-field strict object). A `superRefine` enforces exactly-one-of `prompt`/`messages`; `schema?` is opaque `z.unknown()`; **no credential field is representable** (`packages/contracts/src/gateway/gateway-request.ts:63`). `samplingParams?` (FB.4) carries the optional generation `temperature` (`:72`); `tools?` (TU.1) is the optional `ToolDescriptor[]` allowlist, set ONLY on the `population_generator` route — rule #6 (`:76`).
- `ModelGatewayResponse` = strict `{ accepted, output?, validationResult, providerMeta, langfuseTraceId?, rejection?, toolCallRequests? }` (now 7-field). Refines enforce `accepted ⇔ (validationResult !== 'rejected')` and `rejection` present IFF rejected; no credential field (`packages/contracts/src/gateway/gateway-response.ts:27`). `toolCallRequests?` (TU.1) surfaces the model's requested calls on `finish_reason==='tool_calls'` — `accepted:true` with no `output` but `toolCallRequests` set is a VALID intermediate response (`:39`).
- `ModelRole` = closed 7-union: `population_generator | critic | subtype_check | embedding | final_judge | fusion_synthesis | retrieval` (`packages/contracts/src/gateway/model-role.ts:8`).
- `ModelRoute` = strict `{ role, provider, modelId, capability, fallbackRouteIds[] }`; `provider`/`modelId` are open strings, so embedding can pin OpenAI while critic uses OpenRouter (`packages/contracts/src/gateway/model-route.ts:13`).
- `ProviderCapability` = strict `{ structuredOutputs, embeddings, toolCalling?, streaming? }` — the MVP-lean gate flags (`packages/contracts/src/gateway/provider-capability.ts:8`).
- `ProviderMeta` = shared `{ provider, modelId, gatewayRequestId, tokensIn, tokensOut, costEstimate? }`; **no credential field** (env-only). Returned on every response.

**The TU.1 tool-use contracts** (schema v10, all from `@doppl/contracts/gateway/tool` + the `ChatMessage` union):
- `ToolName` = frozen closed 4-member `z.enum`: `web_search | fetch_url | x_search | youtube_search`. An unlisted tool is unrepresentable — the allowlist by closed-enum (rule #3) (`packages/contracts/src/gateway/tool.ts:17`).
- `ToolDescriptor` = strict `{ name: ToolName, description }` — a non-executing offered-tool descriptor; no code-carrying field is representable (rule #3 by shape, the `CheckRunnerAdapter` technique) (`tool.ts:29`).
- `ToolCallRequest` = strict `{ id, name: ToolName, arguments }` — the model's requested call; `arguments` is the RAW provider JSON-arg STRING kept as DATA, never interpolated as code (rule #5) (`tool.ts:43`).
- `ChatMessage` = a union of three strict variants: the chat message `{ role: ChatRole, content }`, an assistant-tool-call echo `{ role:'assistant', content, toolCalls[≥1] }`, and a tool-result `{ role:'tool', toolCallId, toolName, content }`. The two tool variants widen the `messages` element WITHOUT widening the closed 3-member `ChatRole` (still `system`/`user`/`assistant`) (`gateway-request.ts:49`).
- `CURRENT_SCHEMA_VERSION` = 10 (the sv9→10 TU.1 bump; additive — readers accept `schemaVersion ≤ current`, so v1–v9 envelopes still validate) (`packages/contracts/src/version.ts:51`).

**What it expects from others:** an injected `providerCall: (request) => Promise<ProviderResult>` and a `capabilityFor` resolver (`gateway.ts:43-49`). Adapters expect a `ModelRegistry` for route resolution. The boot root (`apps/api/src/main.ts`) supplies the live clients and registry; tests inject fakes.

**Internal seam types** (not contracts): `ProviderResult = { output: unknown; providerMeta }` (`structured-output.ts:20`) and `ProviderCallFn` (`:26`) are the adapter↔gateway interface.

## Data & state

This layer is **almost stateless** — it holds no run state and writes to no database. The data structures it works with:

- **`RegistryConfig`** (in-memory, built once at boot) — a strict object keyed by all 7 roles, each a `RouteConfig` `{provider, modelId, capability, fallbackRouteIds?}` with **no credential field** (`apps/api/src/model-gateway/config.schema.ts:16`). The default values live in `apps/api/src/config/model-registry.config.ts:18` (OpenRouter cheap tier for generation/critic/subtype_check, strong tier for judge/fusion, direct-OpenAI for embedding, `web-search` for retrieval).
- **`MODEL_ROUTE_OVERRIDE_ALLOWLIST`** (frozen const) — per-role permitted override targets; `population_generator` + `fusion_synthesis` only, `final_judge` deliberately absent (`apps/api/src/config/model-route-allowlist.config.ts:16`).
- **`DEFAULT_PRIOR_ART_CORPUS`** (static const) — the operator-curated retrieval fallback, a list of `{label, snippet, uri?, keywords[]}` entries carrying no secrets (`apps/api/src/config/prior-art-corpus.config.ts:11`).
- **Credentials** — read only from injected `env` at the client factories and at `assertProviderCredentials`; closed over inside the SDK client and never returned. Required vars: `OPENROUTER_API_KEY`, `OPENAI_API_KEY`, `DATABASE_URL` (`registry.ts:19`). Ollama is keyless (`OLLAMA_BASE_URL` only).

The authoritative data this layer *produces* is the `ModelGatewayResponse` — which the **kernel** then turns into events (`energy.spent`, `provider_call_failed`, etc.). The vector/dimension an embedding call returns is persisted by selection into `novelty.scored` (rule #7 / lesson §13), not by this layer.

## How it works (flow)

A single `gateway.call(request)`:

```
domain code            createGateway              adapter / withRetry          provider
  │  call(request)        │                            │                          │
  ├──────────────────────►│  resolveSchema(request)    │                          │
  │                       ├── providerCall(request) ──►│  resolve route + cap     │
  │                       │                            ├── withRetry(attempt) ───►│  (≤1+retries, +1 fallback)
  │                       │                            │◄── raw output ───────────┤
  │                       │◄── {output, providerMeta} ─┤                          │
  │                       ├── applyStructuredOutputDiscipline                      │
  │                       │      safeParse(output)                                │
  │                       │        ok → accepted(parsed.data)                     │
  │                       │        fail + nonempty → ONE repair → re-validate     │
  │                       │        else → rejected(reason)                        │
  │◄── ModelGatewayResponse ──────────────────────────────────────────────────── │
```

1. **Resolve the schema.** `createGateway` narrows the opaque `request.schema` to a `ZodType` via a duck-typed `safeParse` check (`gateway.ts:61`). No schema → the no-validation path: accept the raw output as-is (`gateway.ts:73`).
2. **Call the provider.** `providerCall` resolves `route` + `capability` from the registry, then runs `withRetry`: up to `1 + maxRetries` (default 2) primary attempts, each raced against a per-attempt timeout, then exactly one fallback-route attempt (`retry.ts:96`, `openrouter.adapter.ts:148`). A success returns `{ output, providerMeta }` with the ACTUAL provider/model/tokens; a terminal exhaustion throws a **0-token** `ProviderCallError` (`openrouter.adapter.ts:199`).
3. **Validate.** `applyStructuredOutputDiscipline` does `schema.safeParse(rawOutput)` (`structured-output.ts:115`):
   - **accept** → return `{ accepted:true, validationResult:'accepted', output: parsed.data }` (the parsed/normalized value, lesson §18).
   - empty/null output → **straight reject**, no repair (`structured-output.ts:122`).
   - otherwise **one repair**: build a repair request whose instruction names the schema errors (system message) and carries the invalid output as `wrapUntrusted(...)` DATA in a user message (`structured-output.ts:80`), re-call, re-validate once → `repaired` or `rejected` (`:128-134`). The single `await` is the structural ≤1 bound.
4. **Map failures.** Back in `createGateway`, a caught `ProviderCallError` becomes `{ accepted:false, validationResult:'rejected', providerMeta, rejection:{reason} }` — a failed *call*, not a failed run; any non-provider error re-throws (it's a real bug) (`gateway.ts:89-101`).

**Live dispatch** (`createLiveGateway`, `live-gateway.ts:114`): the injected `providerCall` first short-circuits the `embedding` role to the direct-OpenAI adapter (the d287675 fix — embeddings are a different provider+endpoint than chat), then resolves `route.provider` and looks it up in a dispatch map `{ openrouter, openai, 'web-search', ollama? }`. A provider absent from the map → an honest 0-token `ProviderCallError` (no silent fallback). The map **is** the runtime provider allowlist (`live-gateway.ts:101-145`).

## Tool-use surface — gateway & adapter mechanics (TU.1, schema v10)

**Plain version:** the TU.1 amendment lets a `population_generator` agent do its own research before it answers. The gateway's job here is narrow and mechanical: offer a *closed allowlist* of research tools, translate the multi-turn tool conversation to/from the provider, and — when the model asks to call a tool — surface that request as an intermediate response instead of trying to validate a final answer. The *driving* of the loop (re-injecting results, re-asking, budgeting turns) and the *live tool implementations* live one layer up in boot — see [12-tool-use-research.md](12-tool-use-research.md). This layer owns four pieces:

**1. The tool allowlist (rule #3).** `TOOL_REGISTRY` and `TOOL_IMPLS` are two frozen, deep-frozen parallel maps keyed by the closed 4-member `ToolName` — the descriptor map (what's offered) and the executor-impl map (the code). `resolveTool(name)` is the fail-safe gate: an own-property `hasOwnProperty.call` lookup on BOTH maps, so a name present in one but not the other — or an adversarial `__proto__`/`constructor` — resolves to `{ ok:false, reason:'unregistered_tool' }` and **never executes** (`registry.ts:236`). This is the exact analog of the check-runner allowlist (lesson §11/§39): the descriptor carries no code by shape, the impl lives in the parallel frozen map, and there is no runtime-register path. `offeredToolDescriptors()` (`registry.ts:227`) is the descriptor set a `population_generator` request offers.

**2. The pure SSRF gate for `fetch_url` (rule #3).** `assertSafeFetchUrl(raw)` (`ssrf.ts:108`) is a pure, synchronous, fail-closed FIRST layer that runs before any outbound request: `http(s)` only (rejects `file:`/`ftp:`/`data:`/…), no embedded userinfo credentials, and no loopback/private/link-local/CGNAT/metadata host — across IPv4 AND IPv6, including IPv4-mapped (`::ffff:…`) and NAT64 (`64:ff9b::…`) forms and the `169.254.169.254` cloud-metadata IP (`isPrivateIpv4Int` `ssrf.ts:43`, `isPrivateIpv6` `ssrf.ts:72`, `isPrivateHost` `ssrf.ts:98`). A non-IP hostname passes this literal layer by design — it is the **second** layer (an injected all-records DNS-publicness check + connection pinning that closes the resolve→connect TOCTOU) that catches a public name resolving to a private IP, and that second layer lives in boot/`toolSeams` (12 owns it). The `fetchUrlExecutor` (`registry.ts:105`) fails CLOSED unless BOTH the `httpGet` and `resolveHostIsPublic` seams are wired, and never throws (returns `ok:false` DATA).

**3. The shared OpenAI-protocol message mapper (rule #9).** `toProviderMessages` (`message-mapping.ts:42`) maps the contract `ChatMessage` union to OUR vendor-free provider shape: the assistant-tool-call echo → `{ role:'assistant', content, tool_calls:[…] }`, the tool-result → `{ role:'tool', tool_call_id, content }`. ONE mapper serves BOTH the OpenRouter and Ollama adapters (`ollama.adapter.ts:103`; `openrouter.adapter.ts:111` via `buildParams`) — no vendor SDK type leaks into the surface (lesson §5, extract-at-the-2nd-consumer).

**4. The tool-call branch in the discipline.** In `createGateway` (`gateway.ts:77`), if the provider's result carries `toolCallRequests`, the gateway returns `{ accepted:true, validationResult:'accepted', providerMeta, toolCallRequests }` **without** running `applyStructuredOutputDiscipline` — there is no final answer to validate yet (the `accepted ⇔ ¬rejected` refine still holds; an `accepted:true` response with no `output` is a valid INTERMEDIATE response). The OpenRouter adapter feeds this branch: `buildParams` offers the request's tools as provider function tools, with the parameter JSON-schema sourced from `TOOL_REGISTRY` (a descriptor not in the registry is dropped — `openrouter.adapter.ts:141`); `mapSdkResponse` surfaces tool calls only when `finish_reason==='tool_calls'` and allowlist-filters each via `ToolName.safeParse` so a hallucinated/unlisted tool is dropped (`parseToolCall` `:163`, the filter `:170`, the surface `:306`). The Ollama adapter gained the same shared message-mapping but does not yet offer tools.

**Why the discipline still holds for final answers.** The tool-call branch is the ONLY path that skips the discipline, and it's reachable only on a tool-call turn. The eventual final candidate the model returns (`finish_reason==='stop'`) carries no `toolCallRequests`, so it flows through the unchanged validate / repair(≤1) / reject path exactly as before. And because `tools?` is only ever set on the `population_generator` request, a critic/judge call can never reach this branch (rule #6).

## Design decisions & rationale

- **One port, many adapters** (`ARCHITECTURE.md:248`). Domain code names a *role*, never a provider/model, so providers are swappable and the kernel stays vendor-free and replay-safe. The port's I/O *are* the frozen contracts (lesson §20 — "subsystem seam over frozen contracts").
- **Provider strict structured-output relaxed to `json_object`** (PD.13, `ARCHITECTURE.md:255`, LESSONS §98). A discriminated-union schema (root `anyOf`/optional) is rejected by OpenAI's strict `json_schema` subset (HTTP 400, curl-confirmed under both strict modes — the subset requires a root `object` with every key required). So the adapter requests `response_format:{type:'json_object'}` and conveys the shape as a **trusted, candidate-independent system instruction** derived only from the request schema (`openrouter.adapter.ts:110-138`). Provider strict-mode was only an optimization — the gateway's Zod validate/repair/reject stays authoritative.
- **Retrieval never rejects** (P2.7, `ARCHITECTURE.md:253`, lesson §29). Unlike generation/embedding (throw→reject on terminal failure), retrieval falls back to the curated corpus tagged `fallbackSourced` and never throws — rehearsed demo-safety (RISK-004/005). An empty match is valid data, not a failure (`retrieval.adapter.ts:18-24`, `:162`).
- **Embeddings pinned to direct OpenAI** (`ARCHITECTURE.md:251`). `text-embedding-3-small` is a different provider+endpoint than OpenRouter chat, so the `embedding` role short-circuits before provider-dispatch (`live-gateway.ts:118`).
- **Second live provider (Ollama) is a runtime, not contract, change** (FB.1, lesson §101). `provider` is an open string, so adding Ollama only made `createLiveGateway` provider-dispatching — no contract bump. Ollama is keyless by design (`ollama.adapter.ts:27-29`).
- **Per-run override is clamp-as-hint** (FB.2, `ARCHITECTURE.md:257`, lesson §102), mirroring how caps are clamped — the run *proposes*, the kernel *bounds*.
- **Energy/secrets/events are not the gateway's job** (lesson §23). Keeping the gateway side-effect-free makes the discipline a pure, testable function and keeps the persistence/accounting boundaries single-sourced elsewhere.

## Safety & invariants

This layer is a primary enforcement site for five of the nine load-bearing rules. Each is enforced by *mechanism*, not prompt:

- **Safety rule #9 — provider SDKs live ONLY in adapters, behind the port.** The `ModelGateway` port is *type-only* and its I/O are the frozen contracts, so no vendor type can leak into the port surface (`port.ts:8-14`). Each vendor SDK (`import OpenAI`) appears in exactly one place — its client factory: `createOpenRouterClient` (`openrouter.adapter.ts:250`), `createOpenAIEmbeddingClient` (`openai-embedding.adapter.ts:177`); Ollama uses raw `fetch` confined to `createOllamaClient` (`ollama.adapter.ts:217`). Every adapter exposes an *our-interface* client (`OpenRouterClient`, etc.), never the SDK type. Mechanically backstopped by the forbidden-pattern grep `from ['"](openai|@anthropic-ai|openrouter)` over domain/runtime diffs (`apps/api/CLAUDE.md` forbidden-patterns).
- **Safety rule #5 — model output is untrusted until schema-validated; candidate text is data, not instructions.** `applyStructuredOutputDiscipline` is *the* authoritative validator: `safeParse` → accept(`parsed.data`)/repair(≤1)/reject (`structured-output.ts:110`). An unvalidated output never returns as accepted. The single repair carries the invalid output via the **frozen** `wrapUntrusted` sentinel-wrap in a *user* message, with the correction instruction in a separate *system* message — so injection text in a malformed output ("ignore the schema") is data-to-correct, never an instruction (`structured-output.ts:80-95`). The ≤1 bound is structural (a single `await repair(...)`). Pinned by `apps/api/test/unit/model-gateway/structured-output.test.ts` (`test_repair_does_not_multiply`, `test_candidate_text_is_data_not_instruction`).
- **Safety rule #4 — provider keys load from env only, never threaded into persisted objects.** `RouteConfig`/`RegistryConfig` are `strictObject`s with no credential field — a key is structurally unrepresentable in the config (`config.schema.ts:16`, lesson §27). `assertProviderCredentials(env)` reads injected env and fails fast naming the *var* not the value (`registry.ts:30`). Client factories close the key over the SDK instance and never return it; errors name the var (`openrouter.adapter.ts:252`). `ProviderMeta`/`ModelGatewayRequest`/`ModelGatewayResponse` carry no credential field, so a key cannot ride a request/response/persisted payload.
- **Safety rule #1 — `modelRouteOverride` is allowlist-clamped at two gates and excludes `final_judge` (per rule #6).** Gate 1: `modelRouteOverrideViolation` returns the first non-permitted entry → `POST /runs` 422 *before* the `run.configured` append (`model-route-override.ts:54`). Gate 2: `createRegistryOverlay` re-clamps at resolution — a permitted entry overrides, anything else (including a non-permitted entry that bypassed the route) falls back to the base route (`model-route-override.ts:95-108`). `final_judge` has no allowlist entry, so it is *always* a violation (`reason: 'judge_not_overridable'`, `:66`) — the held-out judge's model can't be swapped, protecting the rule #6 fitness anchor. Pinned by `apps/api/test/unit/model-gateway/model-route-override.test.ts` + `apps/api/test/integration/routes/model-route-override.test.ts`.
- **Safety rule #3 — no arbitrary code execution (the tool allowlist).** Tools are gated like the check-runner registry: `ToolName` is a frozen closed 4-member enum (`tool.ts:17`), `TOOL_REGISTRY`/`TOOL_IMPLS` are frozen parallel maps, and `resolveTool` does an own-property lookup on BOTH, so an unregistered/unimplemented/`__proto__` name resolves to `unregistered_tool` and never executes (`registry.ts:236`). The OpenRouter adapter additionally drops any returned `tool_call` whose name fails `ToolName.safeParse` (`openrouter.adapter.ts:170`). Descriptors carry no executable field (`ToolDescriptor` is a `strictObject`). For `fetch_url`, the pure `assertSafeFetchUrl` is the literal SSRF gate (`ssrf.ts:108`); the DNS-publicness + connection-pinning second layer that closes the resolve→connect TOCTOU lives in boot/`toolSeams` ([12-tool-use-research.md](12-tool-use-research.md)).
- **Safety rule #6 (supporting) — tools attach ONLY to the `population_generator` route.** `ModelGatewayRequest.tools?` is optional and the verifier's critic/judge isolation chokepoint never sets it, so a critic/judge request never carries tools → never gets `toolCallRequests` back → the held-out-judge path is byte-identical to pre-sv10. (The structural wiring gate — `createToolOrchestratingGateway` is wired only as the `population_generator` gateway — lives in boot; see [12-tool-use-research.md](12-tool-use-research.md).)
- **Safety rule #8 (success-only energy) — supporting mechanism.** Adapters do *no* energy accounting; a terminal failure throws a **0-token** `ProviderCallError` (`openrouter.adapter.ts:199`, `ollama.adapter.ts:182`, `live-gateway.ts:126`), and the discipline emits/debits nothing (`structured-output.ts:14`). A failed call therefore cannot produce a debit — the kernel debits success-only.
- **Safety rule #7 (replay) — supporting mechanism.** The live path is never reached on replay; the recorded fake is the replay path (`live-gateway.ts:48`). `createRegistryOverlay` is pure, so replay re-derives the effective route from the persisted override with no provider call (`model-route-override.ts:91`).

## Gotchas & sharp edges

- **The gateway throws for non-provider errors.** `gateway.ts:101` re-throws anything that isn't a `ProviderCallError`. The "never throws" contract covers *provider* failures only; a genuine bug (e.g. a thrown non-error) still surfaces.
- **A request with no schema is accepted raw.** The no-schema path (`gateway.ts:73`) returns the provider output verbatim with `validationResult:'accepted'`. The `embedding` and `retrieval` roles rely on this (they carry no output schema). Callers that *want* validation must supply a `schema`.
- **`structuredSchemaInstruction` calls `z.toJSONSchema(schema)`** (`openrouter.adapter.ts:136`, `ollama.adapter.ts:97`). A schema Zod cannot convert to JSON Schema would throw inside `buildParams` — that throw is *not* a `ProviderCallError` and would propagate. Not observed in tests, but worth noting. **UNVERIFIED:** whether every schema actually passed in (`CandidateContent`, etc.) round-trips cleanly through `z.toJSONSchema`.
- **DRIFT (harmless):** `ARCHITECTURE.md:264` labels the Direct-OpenAI node "embeddings **+ fallback**", implying OpenAI is a generation fallback. In code, the `openai` provider in the live dispatch map routes to the *embedding* adapter when present, else the OpenRouter call as a back-compat path (`live-gateway.ts:103`); there is no OpenAI generation fallback wired. The arch diagram is aspirational here.
- **Stale count elsewhere (not this layer):** some sibling modules (`apps/api/src/runtime/heartbeat.ts:12`, `routes/runs.ts:31`) still cite a "closed 36-member RunEventType" in comments, while the current registry is **41 members** (`apps/api/CLAUDE.md` `RunEventType` row). The model-gateway files do *not* carry this stale count; noted only so a reader cross-referencing those comments isn't misled. Harmless.
- **Live embedding is wired but the legacy-misroute fallback still exists.** If `createLiveGateway` is built *without* an `embeddingClient`, an `embedding` call falls through to provider-dispatch and (with the `openai` provider) hits the OpenRouter chat path — the original always-fails misroute. The boot root always supplies the embedding client (`main.ts:157`), but a hand-built live gateway omitting it silently regresses (`live-gateway.ts:55-60`).
- **The fake gateway's per-role fixtures are NOT the real contracts.** `stub/fixtures.ts:35` uses simplified per-role schemas (e.g. `{ idea: string }` for population) so the *discipline* runs and accepts; they are a seam contract for dependent tracks, not the production `CandidateContent` shapes (`fixtures.ts:11-15`).
- **`selectGateway` honest-throws on a live misconfig.** `useStub:false` with no `liveDeps` throws rather than silently falling back to the recorded fake — a `DOPPL_GATEWAY=live` misconfiguration must fail loud (`stub/fake-gateway.ts:97`, lesson §90).
- **Retrieval has no vendor pinned.** The live-search seam is `client?: undefined` until the §6 retrieval spike wires one; until then the curated corpus is the always-available path (`retrieval.adapter.ts:50-54`). The retrieval credential is *not* in `REQUIRED_CREDENTIAL_ENV` (lazy, curated path needs none — `registry.ts:18`).
- **A tool-call turn deliberately skips the discipline.** `gateway.ts:77` returns `accepted:true` / `'accepted'` with NO `output` when `toolCallRequests` is set — a valid INTERMEDIATE response, not a final answer. A caller that assumes `accepted ⇒ output` would read `undefined`; the boot tool-orchestrating gateway (see [12-tool-use-research.md](12-tool-use-research.md)) is the intended caller and re-asks. Only this branch skips validation; the eventual final answer still runs validate/repair/reject.
- **`fetch_url` fails CLOSED unless both seams are wired.** The literal `assertSafeFetchUrl` gate passes every non-IP domain by design, so the injected `resolveHostIsPublic` DNS check is the ONLY defense against a public name that resolves to a private IP; `fetchUrlExecutor` returns `tool_unavailable` if either `httpGet` or `resolveHostIsPublic` is missing (`registry.ts:113`) — it never silently skips the rebinding check.
- **DRIFT (low, confirmed): the `ToolExecutorDeps` docstring is stale on the SSRF TOCTOU.** `registry.ts:33-38` still describes the connection-pinning resolve→connect TOCTOU as an OPEN "DOCUMENTED MVP RESIDUAL" and says `httpGet` "DISABLES redirect-following", but the actually-wired boot seam (`createSafeHttpGet`/`createPinnedFetch`, `apps/api/src/boot/toolSeams.ts`) pins the socket to the validated IP, re-validates each redirect hop, and declares the TOCTOU CLOSED. No live-path risk (the wired path is pinned + redirect-safe); the in-code annotation just misleads a future rule-#3 auditor. The boot seam is owned by [12-tool-use-research.md](12-tool-use-research.md). *(Flagged, not fixed.)*
- **DRIFT (low, confirmed): `ARCHITECTURE.md` §6 prose doesn't describe the tool subsystem.** The TU.1 *wire* contract IS in the Appendix-A tables (the `ModelGatewayRequest`/`Response` rows + `RunEventType` + sv10), but the §6 *prose* body carries only a generic `toolCalling` capability flag (the "Capability matrix (MVP-lean)" line) — there is no prose description of `TOOL_REGISTRY`/`TOOL_IMPLS`, the two-layer SSRF defense, the tool-orchestrator, or the 3 research-tool impls. Mechanism implemented, undocumented-at-prose. *(Flagged, not fixed.)*
