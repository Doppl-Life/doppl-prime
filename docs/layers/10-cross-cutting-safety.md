# Cross-cutting: Safety & Trust Boundaries

## Executive summary

Doppl breeds AI-generated ideas under selection pressure — an evolutionary loop where LLM-driven "agenomes" generate candidate ideas, critics and checks review them, and a held-out judge scores them. The danger in any such system is that the *agents* (which are just text from a language model) could talk their way past the rules: ignore a budget, rewrite their own score, smuggle instructions into data, or leak a secret into a log. This layer is the answer to that danger. It is not a single file — it is a *synthesis* of nine load-bearing safety rules that are each enforced **mechanically, in code the agents cannot influence**, never by a sentence in a prompt that a model could ignore or be tricked into overriding.

The guiding idea (the project's architecture sentence): *it is not the agent — it is the kernel that breeds the agents; the event log is the truth, and the held-out judge is the floor the organism cannot lift.* Every rule below follows the same pattern: the limit lives in the **kernel** (deterministic TypeScript or a Postgres trigger), the **event log** is the append-only source of truth, and the **judge/rubric** is a frozen anchor. A smart non-expert can read this as: "here are the nine promises Doppl makes, and here is the exact line of code that keeps each one — plus the test that proves it."

This document owns the *cross-layer view* of safety. The individual layers (kernel, event-store, gateway, verifier, selection) each implement their slice; here we show how they compose into nine guarantees and where each guarantee actually lives.

Two new agent-facing capabilities have since landed *under* these same nine rules — agent **tool-use / web-research** (see [12-tool-use-research.md](12-tool-use-research.md)) and the **shared knowledge space** / stigmergy (see [11-shared-knowledge-space.md](11-shared-knowledge-space.md)). Neither adds a tenth rule: each is a new *surface* on existing rules — a kernel-enforced tool-call cap (rule #1), a closed tool allowlist that is the analog of the check-runner allowlist (rule #3), two more untrusted-data channels (tool results + retrieved notes, rule #5), and a comparative (peer-context) held-out judge that stays a runner-computed floor (rule #6). They are folded into the per-rule mechanisms below.

## Interfaces & contracts

This layer exposes **no new public API** — it documents how existing surfaces enforce safety. The key contracts (all in `packages/contracts`) that carry safety by *shape*:

- **`RunCaps`** — strict 6 positive-int caps (`maxPopulation`, `maxGenerations`, `energyBudget`, `maxSpawnDepth`, `maxToolCalls`, `wallClockTimeoutMs`). The cap functions take **only** this object plus consumed/requested counts — no agenome trait or `spawnBudget` is in the signature, so nothing can widen a cap by shape (rule #1). See `apps/api/src/runtime/caps/capEnforcer.ts:35`.
- **`RunEventEnvelope`** — the 14-field append envelope. The append path validates against it *minus* `sequence` and `occurredAt` (`AppendInputSchema = RunEventEnvelope.omit({sequence:true, occurredAt:true})`, `apps/api/src/event-store/append.ts:25`) — so a caller cannot set the log's clock or ordering. `sequence` is the sole ordering key.
- **`CheckRunnerAdapter`** — a `z.strictObject` of pure descriptor fields (`id`/`checkType`/`subtype?`/`label?`). A code-carrying field (`exec`/`command`/`handler`/`fn`/`script`/`code`) is **unrepresentable** → rule #3 by shape (`packages/contracts/src/checks/check-runner-adapter.ts:15`).
- **`criticInput`** — strict 2-field shape: trusted `rubric` vs untrusted `candidate` as *distinct* fields; plus `wrapUntrusted(text)` and the public `CRITIC_INPUT_SENTINEL` (rule #5).
- **`FinalJudgeRubric`** — strict 4-field anchor: closed `FinalJudgeAxis` (5 members) + `immutableToAgents: z.literal(true)` + required `policyVersion` + no authority field (rule #6).
- **`EnergyEvent`** — strict 10-field, `eventType` closed to `llm`/`tool`/`spawn` (no failure member) and `unit: z.literal('doppl_energy')` — rule #8 by shape.

Expected from others: callers (the kernel loop, the worker) supply the `consumed` tallies for caps and the persisted RNG/embedding outcomes for replay; the boot layer injects loaded `process.env` secret *values* (never read inside the pure scrub).

## Data & state

- **`run_events` (Postgres)** — the **only** authoritative state. Append-only (triggers below), per-run monotonic gapless `sequence` (the sole ordering key), `occurred_at` DB-stamped UTC. Every other piece of state (current-state, lineage, SSE feed, run-health, Langfuse, Neo4j) is a **derived projection**, rebuildable from this log, never authoritative (rule #2).
- **`unique(run_id, sequence)`** index + `allocateSequence` via `pg_advisory_xact_lock(hashtext(run_id))` (`apps/api/src/event-store/sequence.ts:19`) — closes the TOCTOU on concurrent same-run appends so the sequence stays gapless.
- **In-code frozen constants** — `DEFAULT_JUDGE_RUBRIC` (`apps/api/src/verifier/judge/rubric.ts:96`, `deepFreeze`'d), `CHECK_RUNNER_REGISTRY` / `CHECK_RUNNER_IMPLS` (`Object.freeze`'d, `apps/api/src/check-runners/registry.ts:102`/`:186`), the route allowlist (`apps/api/src/config/model-route-allowlist.config.ts`). These are *source*, not runtime-writable files — the strongest "no agent-writable path."
- **Energy state** — held as a pure fold (`cumulativeSpend`) over `energy.spent` events, not a mutable counter; the cap enforcer reads the fold of **actual** spend.

## Dependencies

- **Depends on:** `packages/contracts` — all the shape-level safety (strict objects, closed enums, literal flags, the redaction primitive, the isolation primitive). The contracts are *frozen*; downstream code imports, never redefines them.
- **Depends on:** Postgres (Drizzle) — the append-only triggers and the advisory-lock sequence are DB mechanisms, not app code (rule #2 cannot be bypassed by a buggy app caller).
- **Used by:** every other layer. The kernel (`03-runtime-kernel.md`) calls the caps/kill/energy/RNG mechanisms; the event-store (`01-persistence-event-store.md`) is the append + replay boundary; the gateway (`02-model-gateway-providers.md`) carries the structured-output discipline + provider isolation; the verifier (`04-verifier-council-checks.md`) uses the isolation chokepoint, the allowlist, and the immutable judge; selection (`05-selection-scoring-reproduction.md`) consumes the judge anchor read-only; observability (`09-observability.md`) mirrors the redaction at the second (Langfuse) boundary.

## How it works (flow)

The nine rules compose along the request → kernel → gateway/event-store path drawn in `ARCHITECTURE.md` §14:

```
Browser ──REST validate + idempotency──▶ API
   │
   ▼  state-machine + sequence guards
 Kernel ──capability + schema + candidate-as-DATA──▶ ModelGateway ──caps·retry·repair/reject──▶ Providers
   │                                                      ╎ redaction
   │ append-only + txn + redaction                        ▼
   ▼                                                    Langfuse
 Event store (run_events)  ──derived-only──▶ Projections / UI / Neo4j
   │
   ▼ allowlist adapters, no arbitrary code
 Check runners
```

The single authoritative **append path** (`apps/api/src/event-store/append.ts:62`) is where rules #2 and #4 meet in one transaction:

1. Validate the envelope (strict, minus server fields) — `append.ts:66`. A schema-invalid envelope rejects with nothing written.
2. Per-type narrow + payload-DoS ceiling — `append.ts:78`. Returns the *parsed* payload (so a pre-transform value can't reach the log).
3. **Secret-redaction scrub** on the validated payload — `append.ts:85` (rule #4).
4. Allocate the gapless per-run `sequence` under the advisory lock — `append.ts:88`.
5. Insert; the DB stamps `occurred_at` — `append.ts:91`. The store exposes only `append` + `readByRun` — never update/delete (rule #2).

**Replay** (`apps/api/src/event-store/replay-reader.ts:37`) reads that log back, validates ordering (out-of-order before gap, then schema-too-new), and folds — calling **no provider** (rule #7, structural: the module imports no gateway/embedding/web seam).

## Design decisions & rationale

- **Kernel-enforced, never prompt-enforced (the whole thesis).** A prompt is text a model can ignore or be injected with; it can never hold a limit. So every cap, the kill switch, the energy ledger, and the spawn clamp are *pure decision functions* in the runtime kernel that the loop calls — they emit nothing and read no agent-controllable input (`ARCHITECTURE.md` §5, root `CLAUDE.md` rule #1; lesson §48). The forbidden-pattern list in `apps/api/CLAUDE.md` even greps the staged diff for cap-in-prompt and provider-SDK-in-domain.
- **Structural over runtime-guarded.** Where possible the rule is made *unrepresentable* rather than checked: a `z.strictObject` with no code field (rule #3), `z.literal(true)` for the judge flag (rule #6), no failure member in `EnergyEventType` (rule #8), an import-list with no provider seam on the replay path (rule #7). A structural pin can't be forgotten at a call site (lessons §9, §11, §30).
- **Event log is the single truth; everything else is derived.** Locked in `ARCHITECTURE.md` §9 / §19 (Postgres append-only, not current-rows/SQLite/Neo4j-as-truth). This makes replay-determinism and auditability possible and means a projection bug can never corrupt history.
- **Two redaction boundaries, mirrored exactly.** The scrub runs before append *and* before Langfuse emit (`ARCHITECTURE.md` §14; lesson §52) — the second boundary mirrors the first's full discipline (ceiling-then-scrub, env-value layer), never a thinner copy.
- **MVP posture, but safety is never cut.** This is a 2-week capstone, yet `ARCHITECTURE.md` §14/§5/§4 keeps all nine invariants load-bearing. Deferred items (hosted access gate, least-privilege DB role, per-version upcasters) are scope, not safety relaxations — flagged below.

## Safety & invariants

All nine, each with the one-line invariant, **where it lives**, **how it is mechanically enforced** (kernel/structural, never prompt), and the **pinning test**.

### Rule #1 — Caps are kernel-enforced, never prompt-enforced
*Population / generations / energy / depth / tool-calls / wall-clock caps + kill switch live in the kernel; `spawnBudget` is a hint clamped to `min(remaining caps)`.*
- **Where + how:** `enforceCap` (`apps/api/src/runtime/caps/capEnforcer.ts:35`) is a **pure fail-closed** decision — `consumed + requested > cap` is denied; the cap reads **only** from the `RunCaps` argument, and the signature takes no trait/`spawnBudget`, so nothing can raise a cap by shape. `enforceWallClock` (`:55`) uses an injected `elapsedMs` (no ambient clock). `clampSpawnBudget` (`apps/api/src/runtime/spawn/spawnBudgetClamp.ts:22`) returns `min(spawnBudget, max(0, remainingPopulation))` — the hint can only *narrow*. The kill switch (`apps/api/src/runtime/caps/killSwitch.ts:111`) maps any breach/operator-stop to a §3-legal terminal, each transition validated through the state-machine guards. The REST route additionally rejects an above-ceiling cap override 422 (`POST /runs`), but the kernel is the *sole* authority (the demo cap-override defers to it — lesson §89).
- **New surface (tool-call cap relay).** Agent tool-use is bounded by the same kernel mechanism. When the loop calls the generation gateway it passes `toolBudget = max(0, maxToolCalls − toolCallsConsumed)` (`apps/api/src/runtime/loop/generationLoop.ts:713`) — but that is a *clamped hint* a tool-orchestrating gateway self-limits to; it can never hold the cap. The **authoritative backstop is the inline `enforceCap('maxToolCalls')` relay** (`generationLoop.ts:722`): before each surfaced tool call is relayed/debited the loop reserves a slot, the `(cap+1)`th call is **denied** (not relayed, not finished, not debited), and a `cap_breach` latches → the post-batch drain halts the run. The capture+check+increment is synchronous (no `await` between read and `+= 1`), so it stays atomic across concurrent agenome tasks. Confirmed gotchas live in *Gotchas & sharp edges* below: (a) a **within-batch energy overshoot** ~the concurrency ceiling under concurrent agenome generation (the generation-boundary kill is the authoritative enforcer); (b) an **execution-vs-accounting gap** where outbound tool IO can exceed `maxToolCalls` under concurrency while the *recorded/debited* ledger stays capped — accounting is never bypassed (medium sev, not a rule-#1 accounting hole).
- **Test:** `apps/api/test/unit/runtime/caps/capEnforcer.test.ts:33` (`enforce_cap_fails_closed_over_ceiling`), `killSwitch.test.ts`, `spawnBudgetClamp.test.ts` (lesson §48/§80).

### Rule #2 — The event log is append-only and authoritative
*Postgres `run_events` (per-run monotonic `sequence` = sole ordering key) is the source of truth; no in-place historical edits; every projection is derived and rebuildable.*
- **Where + how:** **DB triggers**, not app discipline. `run_events_append_only` (`apps/api/src/event-store/migrations/0001_run_events_append_only.sql:14`) is a `BEFORE UPDATE OR DELETE` row-level trigger that `RAISE EXCEPTION`s; `run_events_no_truncate` (`:20`) is a `BEFORE TRUNCATE` statement-level trigger (a row trigger can't catch TRUNCATE). The only write path is `append.ts:62`; it omits `sequence`/`occurredAt` from the input schema so the caller can't forge order or clock (`append.ts:25`). `unique(run_id, sequence)` + the advisory-lock allocation (`sequence.ts:19`) keep it gapless under concurrency.
- **Test:** `apps/api/test/integration/event-store/migrations.test.ts:73` (UPDATE/DELETE rejected), `:87` (TRUNCATE rejected), `:91` (duplicate `(run_id, sequence)` rejected), `:100` (`occurred_at` DB-stamped). Append-path discipline pinned in `test/integration/event-store/append.test.ts` (lessons §25/§26).

### Rule #3 — No arbitrary code execution
*Check-runners are an allowlisted registry of non-executing adapters; an unregistered or execution-requiring check returns `check.completed{status:skipped, reason}`.*
- **Where + how:** Pinned **two ways**. (a) *By shape:* `CheckRunnerAdapter` is a `z.strictObject` with no code-carrying field representable (`packages/contracts/src/checks/check-runner-adapter.ts:15`). (b) *By gate:* `resolveCheckAdapter` (`:52`) does an **own-property** lookup (`Object.prototype.hasOwnProperty.call`, defeating `__proto__`/`constructor` bypass), never executes, never throws, and fails safe to a `skipped` CheckResult on an unregistered id. The impls live in a *parallel frozen map* `CHECK_RUNNER_IMPLS` (`apps/api/src/check-runners/registry.ts:186`); a registered descriptor with no impl is recorded `skipped{execution_required}` by `runCheck` (`apps/api/src/check-runners/run-check.ts:107`). The check fns themselves are pure `(input) => CheckResult` — `candidate` is opaque data.
- **New surface (the tool-use allowlist).** Agent web-research uses the *same* allowlist discipline, one layer over. `ToolName` is a **frozen closed 4-member `z.enum`** (`web_search`/`fetch_url`/`x_search`/`youtube_search`, `packages/contracts/src/gateway/tool.ts:17`) — an unlisted tool is unrepresentable. `ToolDescriptor` is a `z.strictObject` with no code-carrying field (rule #3 *by shape*, `tool.ts:29`). The runtime keys two **frozen parallel maps** — `TOOL_REGISTRY` (descriptors, `deepFreeze`'d, `apps/api/src/model-gateway/tools/registry.ts:185`) and `TOOL_IMPLS` (executor impls, `Object.freeze`'d, `:219`) — and `resolveTool` (`:236`) does an **own-property lookup on BOTH** (an adversarial `__proto__`/`constructor` resolves to neither), never executes, never throws, and fails safe to `unregistered_tool`. The OpenRouter adapter additionally drops any returned tool-call whose name fails `ToolName.safeParse`. Descriptors carry **no executable field**.
- **New surface (two-layer SSRF defense for `fetch_url`).** Because `fetch_url` reaches the network, it is guarded twice. *Layer 1* is the pure synchronous `assertSafeFetchUrl` (`apps/api/src/model-gateway/tools/ssrf.ts:108`): http(s)-only, no userinfo creds, and blocks loopback/private/link-local/CGNAT/metadata IPs across IPv4 + IPv6 (incl. mapped/NAT64 forms). *Layer 2* is the injected resolver + connection pinning in the boot seam (`createSafeHttpGet`/`createPinnedFetch`, `apps/api/src/boot/toolSeams.ts:111`): an all-records (every A **and** AAAA) DNS publicness check, then the socket is **pinned to the validated IP** via Node's `lookup` hook with the hostname kept for TLS SNI + Host — **closing the resolve→connect TOCTOU** (rebinding DNS that answered "public" to the resolver cannot be re-resolved to a private IP at connect time), re-validating each redirect hop. `fetch_url` fails **closed** unless *both* `httpGet` and the resolver are wired. ⚠ A stale in-code annotation contradicts the wired behavior — see *Gotchas & sharp edges* (`registry.ts:33`).
- **Test:** `packages/contracts/test/checks/check-runner-adapter.test.ts:21` (no code field), `:71` (`__proto__`/`constructor`/`toString` → skipped), `apps/api/test/integration/check-runners/run-check.test.ts` (lessons §11/§39/§43).

### Rule #4 — Secrets never leave the server
*Provider keys never enter prompts, event payloads, Langfuse traces, or UI payloads — a redaction scrub runs at the persistence boundary (before append AND before Langfuse emit).*
- **Where + how:** The frozen `scrubSecrets` (`packages/contracts/src/security/redaction.ts:121`) applies **value-pattern** matching (`sk-…`, `Bearer …`, `Basic …`, anchored + length-gated) and **sensitive-key** whole-value redaction (`token`/`secret`/`password`/… — except number/boolean, which can't be a credential, so `tokensIn`/`tokensOut` round-trip; lesson §46). The event-store wrapper `scrubEventPayload` (`apps/api/src/event-store/redaction.ts:103`) composes that and adds the **env-value layer** — any payload string/array-element/**key** containing a loaded `process.env` secret value (≥8 chars) is redacted with de-collision (keys matter because `RunEventEnvelope.payload` is open-key `z.record`; lesson §21). It runs at `append.ts:85` *before* the only insert. Credentials are **never threaded into request/response objects** — env-only, fail-fast at boot, structurally unrepresentable in the config object (lesson §27). The same discipline mirrors before Langfuse emit (`09-observability.md`, lesson §52).
- **Test:** `packages/contracts/test/security/redaction.test.ts`, `apps/api/test/unit/event-store/redaction.test.ts` (env-value layer catches a non-pattern DB password under no sensitive key).

### Rule #5 — Model output is untrusted until schema-validated; candidate text is data, not instructions
*Candidate text reaches critics/judges only inside a sentinel-delimited field, never interpolated into an instruction string.*
- **Where + how:** **One chokepoint.** `assembleIsolatedRequest` (`apps/api/src/verifier/isolation/candidate-as-data.ts:51`) builds *every* critic/judge/check request: the trusted instruction + a fixed `ISOLATION_DATA_FRAMING` form the `system` message (byte-identical regardless of candidate), and the candidate rides alone in a `user` message via the frozen `wrapUntrusted` (`packages/contracts/src/verifier/critic-input.ts:57`). `wrapUntrusted` neutralizes any candidate-embedded `CRITIC_INPUT_SENTINEL` so the result holds the sentinel *exactly twice* for any input — an agenome can't forge a delimiter boundary (lesson §8). Separately, the **gateway** treats output as untrusted: `applyStructuredOutputDiscipline` (`apps/api/src/model-gateway/structured-output.ts:110`) validates against the request schema → accept / **repair (≤1, a single `await` bound)** / reject; the repair carries the invalid output as `wrapUntrusted` DATA, never an instruction. An unvalidated output never enters the log.
- **New surfaces (three more untrusted channels, all reusing `wrapUntrusted`).**
  1. **Tool results** re-enter the model as *data*, not instructions: `toolResultMessage` (`apps/api/src/boot/toolOrchestrator.ts:95`) prefixes the trusted `TOOL_RESULT_DATA_FRAMING` and wraps the result via the frozen `wrapUntrusted` in a distinct `role:'tool'` message; the raw provider argument string is kept as DATA (`tool.ts`), parsed only inside `parseArgs` *before* the executor. A tool result is the prime injection vector, so it is the most important new channel.
  2. **Retrieved KB notes** thread into the *population_generator* request as a **second `wrapUntrusted` user message** (`buildPopulationRequest`, `apps/api/src/runtime/loop/generationLoop.ts:87` → the extra user message at `:109`); the only trusted addition is the candidate-independent `KB_RETRIEVAL_FRAMING` in the *system* message, which names the notes as DATA. Absent/empty notes → byte-identical baseline.
  3. The **comparative (peer-context) judge** wraps each candidate in its *own* sentinel-delimited user message via `assembleIsolatedComparativeRequest` (`apps/api/src/verifier/isolation/candidate-as-data.ts:120`) with the candidate-independent `ISOLATION_COMPARATIVE_FRAMING` (`:78`). **New context surface:** N untrusted candidates now share one model context, so candidate A's text is in-context when the model scores B — an injection vector the fully-isolated single-candidate path did not have. It is **bounded/mitigated**: per-blob sentinel-wrap (each blob's sentinel independently neutralized), the explicit data-not-instructions framing, and the **peer-invariant acceptance floor** (rule #6) — a successful injection can at worst move a sibling's *model-output* axis scores within 0–10, the same bound as self-manipulation; the runner-computed acceptance aggregation stays structurally peer-invariant.
- **Test:** `apps/api/test/unit/verifier/isolation/candidate-as-data.test.ts:47` (candidate only in the sentinel-wrapped user message), `:61` (system message byte-identical for different candidates), `packages/contracts/test/verifier/critic-input.test.ts`, `apps/api/test/unit/model-gateway/structured-output.test.ts` (lessons §8/§23/§38).

### Rule #6 — The held-out judge, its rubric, and the scoring policy are immutable to agents
*The bedrock fitness anchor the organism cannot move (anti-reward-hacking).*
- **Where + how:** **Stacked.** *By shape:* `FinalJudgeRubric` (`packages/contracts/src/verifier/final-judge-rubric.ts:38`) is a `z.strictObject` of closed `FinalJudgeAxis` (5 members), `immutableToAgents: z.literal(true)`, required `policyVersion`, and **no** mutation/override/authority field representable. *By load path:* `loadJudgeRubric` (`apps/api/src/verifier/judge/rubric.ts:38`) re-asserts what the schema can't — full-5-axis-set completeness (`:51`) and `immutableToAgents === true` read through an `unknown`-typed local so it survives a future schema relaxation (`:66`); the source is the deep-frozen in-code `DEFAULT_JUDGE_RUBRIC` (`:96`), never an agent-writable path. *By computation:* the model supplies only per-axis scores (now **0–10**, widened from 0–5 in Wave 2 Step 4 — a runtime/scoring concern, *not* a frozen-rubric field; `axisScore = z.number().min(0).max(10)`, `apps/api/src/verifier/judge/judge-core.ts:31`); the **runner** computes the acceptance aggregate (`computeAcceptanceMetric`, `apps/api/src/verifier/judge/judge-core.ts:89`) as a weighted sum over the *rubric's* axes — the model never supplies its own winning number (a permissive `z.object` strips any model-sent `score`/`total`/`acceptance`/`id`, `judge-core.ts:33`). Even the FB.8 per-axis *rationale* is explanatory output that **never** feeds acceptance (lesson §108). The judge sits outside the breeding loop (`final_judge` role, never a rotating critic) and is **excluded from the per-run model-route allowlist** (`apps/api/src/config/model-route-allowlist.config.ts:26`) so its model is not run-swappable.
- **New surface (comparative / peer-context judge).** The judge was hoisted out of the per-candidate loop: council + checks run per-candidate concurrently, then **one `runComparativeJudge` call scores the whole generation side-by-side** (`apps/api/src/verifier/judge/comparative-judge.ts:82`, wired at `apps/api/src/verifier/verify-seam.ts:145`) so the model spreads its scores. Crucially, **acceptance is still RUNNER-computed as Σ(that candidate's axes × the immutable rubric weights) with NO peer term** — `computeAcceptanceMetric` (`judge-core.ts:89`) has no peer parameter, so acceptance is *peer-invariant* given fixed axis scores: a uniformly-weak generation cannot manufacture a high best. The single- and comparative judges share `judge-core` byte-identically (a divergence would be a silent rule-#6 scoring difference). The comparative judge is therefore a true **floor**.
- **New surface (tools + retrieval attach to one route only).** Both new capabilities reach **only the `population_generator` route**, structurally: `createToolOrchestratingGateway` (which alone offers tools) and `createKnowledgeRetriever` are wired as the generation gateway, while *the verify seam calls `modelGateway` directly* for critic/judge (`apps/api/src/boot/composeRuntime.ts:237`). The §38 critic/judge isolation chokepoint has no `tools`/notes parameter, so a critic or judge request can never carry a tool or a retrieved note → the held-out judge anchor is **byte-identical**.
- **Immutability across the diff.** The frozen rubric/policy contracts (`final-judge-rubric.ts`/`judge-result.ts`/`scoring-policy.ts`) are **byte-identical across the sv9→sv10 window** (the schemaVersion bump was the unrelated tool-use TU.1 change; the `FinalJudgeRubric` snapshot is unchanged). The runtime rubric `policyVersion` moved `final-judge-mvp-1` → `final-judge-mvp-3` (`apps/api/src/verifier/judge/rubric.ts:98`) to *record* that the judge's scoring behavior moved (0–5 → 0–10 + comparative) — axes/weights/`immutableToAgents` are unchanged; this is immutability-via-versioning (lesson §12), a developer-controlled record, not an agent-writable change.
- **Test:** `packages/contracts/test/verifier/final-judge-rubric.test.ts:58` (unknown axis rejected), `:65` (`immutableToAgents:false` rejected), `apps/api/test/unit/verifier/judge/rubric.test.ts` (4-axis/6-with-dup rejected), `judge-call.test.ts` (runner-computed acceptance) (lessons §9/§17/§40/§42).

### Rule #7 — Replay calls no providers
*Replay reconstructs from the persisted per-run RNG seed + outcomes and never calls model / embedding / web providers.*
- **Where + how:** **Structural.** The replay reader (`apps/api/src/event-store/replay-reader.ts:37`) is given a narrowed `Pick<EventStore,'readByRun'>` (writes unreachable) and **imports no provider/embedding/web seam** — so "calls no provider" holds by construction, not a runtime guard. It *validates-not-sorts* (out-of-order, gap, schema-too-new throw `ReplayIntegrityError`; never silently re-sorts a corrupted log) and re-folds the persisted log. Expensive-once values are persisted as **required** fields so replay reads them: the RNG `seededRng.ts` derives all sampling from the persisted `RunConfig.rngSeed` (`apps/api/src/runtime/rng/seededRng.ts:55`, mulberry32, no ambient randomness), embedding vectors are required on `NoveltyScore`, RNG outcomes required on `ReproductionEvent`, and the model-route overlay re-applies deterministically (`apps/api/src/model-gateway/model-route-override.ts:95`).
- **New surfaces (all three new capabilities are replay-safe by construction).**
  - **In-run retrieval** persists its *outcome* — the retrieved **note-id SET** rides the existing `candidate.generation_started` op-marker (appended via the append path, no schemaVersion bump). `retrieveNotes` (`apps/api/src/selection/knowledge/retrieve.ts:78`) is **pure** (no gateway/clock/`Math.random`); on replay the research-notes projection **re-folds** that persisted set into `retrieved` edges (`apps/api/src/projections/research-notes.ts:191`) with **no provider or index call**. The boot retriever runs LIVE only — the loop is the live worker path; replay is a projection re-fold. (This is why live nondeterminism under concurrent siblings — which sibling's note landed first — is captured, never re-derived.)
  - **Tool execution** is IO confined to the boot/model-gateway layer (`toolOrchestrator` + `toolSeams`); the replay-pure runtime loop only **relays** surfaced observations and persists `tool_call.started/finished` (truncated-with-marker + scrubbed). The recorded/replay path takes the **pass-through gateway** (`apps/api/src/boot/composeRuntime.ts:243`) and re-reads persisted tool results — never re-fetches/re-searches.
  - **The comparative judge** never re-issues its call on replay: acceptance is recomputed from the persisted per-candidate `JudgeResult.axisScores` via the deterministic `computeAcceptanceMetric` (`judge-core.ts:89`, no random/clock).
- **Test:** `apps/api/test/unit/projections/replay-summary.test.ts:122` (`test_replay_imports_no_provider` — import-ban regex over the transitive path + no `fetch(`/`Math.random(` call), `apps/api/test/unit/runtime/rng/persistOutcomes.test.ts` (lessons §30/§47/§55).

### Rule #8 — Energy = successful productive spend only
*Failed / retried / repaired attempts emit `provider_call_failed` and do NOT debit energy.*
- **Where + how:** **By shape + by math.** The frozen `EnergyEventType` is closed to `llm`/`tool`/`spawn` with **no failure member**, so a failed attempt *cannot* yield an `EnergyEvent` (lesson §49). `reconcileEnergy` (`apps/api/src/runtime/energy/estimateReconcile.ts:68`) derives the llm `actual` **only** from real `providerMeta` token counts — `providerMeta` is required for the llm reconcile (compile-time, plus a fail-loud backstop at `:75`), so an estimate can never leak into the cap-relevant spend. `cumulativeSpend` (`apps/api/src/runtime/energy/energyLedger.ts:44`) folds **actual** (not estimate) per scope, and *only* over `energy.spent` events. The gateway's validate/repair/reject and the markers (`check.started`, `judge.review_started`, …) debit **no** energy. Finiteness comes from bounded retries + per-call timeout + wall-clock cap.
- **New surfaces (only successful productive spend debits).** **Tool energy** is debited per tool call **only when `toolCall.ok !== false`** (`apps/api/src/runtime/loop/generationLoop.ts:766`): a blocked/unavailable/failed tool (`ok:false`) is relayed for observability and counts toward `maxToolCalls` (rule #1) but debits **no** energy. **Retrieval/embedding** emits no energy event — there is no debit around the `retrieveKnowledge` call. **Reproduction** (fusion + mutation slots) and the **concurrent council + checks** fan-out are execution-strategy only and emit **no** `energy.spent`. Every one of these still flows through the closed `EnergyEventType` (`llm`/`tool`/`spawn`, no failure member), so a non-spend simply yields no `EnergyEvent` by shape.
- **Test:** `apps/api/test/unit/runtime/energy/estimateReconcile.test.ts`, `energyLedger.test.ts`, plus `packages/contracts/test/domain/energy-event.test.ts` (no failed/retried/repaired field representable) (lesson §49).

### Rule #9 — Postgres only (no SQLite); provider SDKs only behind the gateway
*Provider SDKs are never imported into domain/runtime modules.*
- **Where + how:** All vendor SDK use is confined to `apps/api/src/model-gateway/adapters/` (openrouter, openai-embedding, retrieval, ollama) + `live-gateway.ts`; domain/runtime sees only the `ModelGateway` port and `ProviderCapability` (the layer dependency direction in `apps/api/CLAUDE.md` is enforced top-depends-on-bottom). The forbidden-pattern grep in `apps/api/CLAUDE.md` warns on `from ['"](openai|@anthropic-ai|openrouter)` outside the gateway, and on raw `run_events` writes. **No SQLite** anywhere (`ARCHITECTURE.md` §19 locked decision; persistence is Drizzle + Postgres). The live gateway dispatches per-provider through an in-runtime allowlist (lesson §101).
- **Test:** the import-direction is a documented boundary lint; the replay import-ban test (rule #7) doubles as a no-gateway-in-replay check. **UNVERIFIED:** I did not find a single dedicated automated test asserting "no provider SDK imported into domain/runtime" or "no `sqlite` dependency" — enforcement here is the forbidden-pattern grep (`/preflight`) + module organization + code review, not a green unit test. (The grep is real; an always-on CI assertion would be stronger.)

## Gotchas & sharp edges

- **There is no `safety/` directory.** Safety is distributed across nine files in five layers. Searching for "the safety module" finds nothing — this doc *is* the index.
- **The kernel decides, the loop emits.** Every cap/kill/energy/clamp function is *pure and emits nothing* (lessons §33/§48/§49). The decision and the event are deliberately split (`ARCHITECTURE.md` §5 ownership). A reader expecting the cap enforcer to also append the breach event will not find it there — the loop does.
- **Redaction is over-eager on purpose, but type-aware.** A sensitive *key* whole-redacts its value — *except* numbers/booleans, because a credential is always a string and redacting `tokensIn` to the string `[REDACTED]` would silently corrupt the append-only log on read (lesson §46, a fixed `[high]`). The fix was type-awareness, not narrowing scope.
- **The env-value layer needs ≥8-char secrets.** A blank or 1–2-char env var would `''.includes`-match everywhere; `MIN_SECRET_LENGTH = 8` (`apps/api/src/event-store/redaction.ts:24`) guards against catastrophic over-redaction.
- **DRIFT (least-privilege DB role):** `apps/api/LESSONS.md` §25 describes rule #2's full mechanism as "DB triggers **AND a least-privilege runtime role** (non-owner/non-superuser; migrations as a separate owner)" because triggers are privilege-defeatable (`session_replication_role` / `DISABLE TRIGGER`). The shipped migration ships only the triggers + unique index — **no `CREATE ROLE`/`GRANT`/`REVOKE` is present** in `apps/api/src/event-store/migrations/` and the migrations test asserts no role separation. So the append-only guarantee is trigger-only in the MVP; a superuser/owner connection could disable the triggers. This is an MVP gap consistent with the prototype posture, but the lesson's "least-privilege role" half is **aspirational, not implemented**.
- **UNVERIFIED (rule #9 automated test):** see rule #9 above — no green test pins "no SDK in domain" / "no SQLite"; it is grep + convention.
- **DRIFT (event-type count, harmless):** the redaction source comment says "30/36 event types fall to the generic open-key schema" (`apps/api/src/event-store/redaction.ts:51`) and `ARCHITECTURE.md` §14 says "30/36"; the registry is now **41 members** (`apps/api/CLAUDE.md` cross-doc table, now schemaVersion 10 after the tool-use TU.1 bump). The *mechanism* is unaffected (most types still use the open-key generic payload), but the "/36" figure is a stale-but-harmless note.
- **DRIFT (stale in-code SSRF annotation — confirmed, rule #3):** the `ToolExecutorDeps` docstring in `apps/api/src/model-gateway/tools/registry.ts:33`–38 still describes the connection-pinning TOCTOU as an *open* "DOCUMENTED MVP RESIDUAL … does NOT yet pin the connection to the validated IP, so a resolve→connect TOCTOU … remains," and says `httpGet` "DISABLES redirect-following." But the actually-wired seam (`createSafeHttpGet`/`createPinnedFetch`, `apps/api/src/boot/toolSeams.ts:111`, with the seam header at `:27`–29 declaring the TOCTOU **CLOSED**) **pins the socket to the validated IP and FOLLOWS redirects with per-hop re-validation.** No behavior risk — the live path is pinned + redirect-safe — but the annotation is **misleading for a future rule-#3 auditor reading the registry**. The annotation is the stale half, not the wiring.
- **DRIFT (ARCHITECTURE.md §7 judge scale — confirmed, medium):** the arch still says "the MVP rubric is a **5-axis 0–5 scale**" (`ARCHITECTURE.md:279`) and the Appendix-A `JudgeResult` row describes "the per-axis 0–5 breakdown" (`:544`), but the code now scores each axis **0–10** (`judge-core.ts:31`, `JUDGE_AXIS_MAX_SCORE=10`, the "INTEGER 0–10" instruction; max acceptance now `10·Σweights=50`). The scale is explicitly a runtime/scoring concern, *not* a frozen-rubric field — so the contract is unaffected, but the arch prose is stale. (A matching stale "0–5" comment lives at `apps/api/src/selection/reproduction/directed.ts:14`; behavior is unaffected — `weakestJudgedAxis` is a scale-agnostic argmin.)
- **DRIFT (ARCHITECTURE.md §7 judge execution mode — confirmed, medium):** the arch describes the held-out judge as applying the rubric **per-candidate** (one judge call per candidate); the code **hoisted it to a comparative peer-context judge** scoring a whole generation in one gateway call (`comparative-judge.ts:82`). The new mode and its rule-#5 cross-candidate context surface (N candidates sharing one model context) are undocumented in the contract. The safety posture is intact (acceptance is peer-invariant; see rule #6), but the arch text predates the hoist.
- **DRIFT (ARCHITECTURE.md §6 tool subsystem prose — confirmed, low):** the TU.1 *wire contract* is captured in the Appendix-A tables (sv10 `ToolName`/`ToolDescriptor`/`ToolCallRequest` + the optional `tools?`/`toolCallRequests?` rows), but the §6 model-gateway **prose** never describes the runtime tool subsystem itself — `TOOL_REGISTRY`/`TOOL_IMPLS`, the two-layer SSRF defense + TOCTOU-closing pinned fetch, the multi-turn tool-orchestrator, or the three research-tool impls. Implemented but undocumented at the prose level.
- **Within-batch energy overshoot (confirmed, low; rule #1).** Under the new concurrent agenome generation (`mapLimit`), each in-flight task re-checks `detectKill` before its generate, but several can pass the check and then all debit — so a single generation's batch can exceed `energyBudget` by up to ~the concurrency ceiling before the next boundary kill fires. The code documents this as bounded/acceptable (the fan-out is clamped to energy headroom so a batch never *starts* more calls than the budget can pay for) and relies on the **generation-boundary kill as the authoritative rule-#1 enforcer**. Strict per-call cap serialization is relaxed versus the prior sequential loop.
- **Execution-vs-accounting gap on tool calls (confirmed, medium; rule #1).** Because the orchestrator's `toolBudget` is a *hint*, outbound tool IO can exceed `maxToolCalls` under concurrency — but the **recorded/debited ledger stays capped**: the inline `enforceCap('maxToolCalls')` relay (`generationLoop.ts:722`) denies the `(cap+1)`th call from being relayed/finished/debited and latches a `cap_breach`. This is an *execution-vs-accounting* gap (a few extra network calls may fire), **not** a rule-#1 accounting bypass — the authoritative count and energy debit are never raised past the cap.
- **Whole-call rejection sinks ALL candidates (confirmed, medium; robustness, not a rule break).** If the single comparative judge gateway output is rejected or un-assemblable, `rejectAll` (`comparative-judge.ts:134`) emits `output_schema_rejected` for *every* candidate that generation → `judgeAcceptance` returns `present:false`/`value 0` for all, so the dominant `judge_acceptance` fitness component loses its separating signal generation-wide. It fails **SAFE** — a `JudgeResult` is never fabricated (rule #6 preserved) — but it is a robustness regression versus the single-candidate path, where one bad output affected only one candidate. The comparative call also sets no `maxTokens`, so a large generation's combined output could truncate → un-assemblable → `rejectAll`, amplifying this.
- **The access gate is a deferred seam, not shipped.** `ARCHITECTURE.md` §14 (Q3) keeps the showcase local-first with no auth build; mutating-endpoint auth is a documented seam, deferred. Don't read the doc as if an auth layer exists.
