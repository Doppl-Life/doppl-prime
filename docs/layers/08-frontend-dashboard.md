# Frontend Dashboard

## Executive summary

This is the web app a person actually watches — the **live observatory** for a Doppl evolution run. It is a React 19 + Vite single-page app (`apps/web/`) with a handful of routes: a home list of runs, a launcher, a 3-pane "organism view" that draws the lineage tree as it grows, a **knowledge-evolution** view that draws the swarm's shared research, and a final-idea payoff screen. Its one rule is that it is **strictly read-only over the backend's projections**: it fetches REST views and listens to a live event stream, but it never owns or mutates authoritative state — the only writes it issues are two idempotent commands (start a run, stop a run), and even those just hand control to the API + kernel.

Everything the dashboard shows is *derived*. A pure reducer folds the run's event stream — ordered strictly by each event's per-run `sequence` number, never by wall-clock time — into a small view state; pure "selector" functions fold the same events into the fitness charts, the energy meters, the critic gauntlet, and the deep per-node telemetry. Because all of it is pure over the persisted events, **replay looks identical to a live run** (`apps/web/src/state/reducer.ts:8`). The lineage graph is a React Flow tree laid out deterministically by a **per-generation column layout** (each generation is a column, read left→right as the run advances), and every status is encoded by shape + icon + label, not color alone, so it survives a projector and a colorblind reviewer (`apps/web/src/components/core/StatusBadge.tsx:6`). A companion **Knowledge-Evolution** graph renders the swarm's accreting shared research (the stigmergy KB) on the same column scheme.

The layer imports **no backend internals** — it shares only the frozen Zod schemas in `packages/contracts`, validates every server payload at the boundary, and never fetches or renders a secret. A smart reader can think of it as: "the backend is the truth; this is a faithful, accessible, replay-identical *mirror* of that truth, plus two buttons."

## Responsibilities

- **Render the run as a live observatory** — the React Flow lineage tree, fitness/generation/energy charts, candidate inspector, critic gauntlet, subtype-check evidence, energy meters, health gauge, activity ticker, and the final-idea proof panel (`apps/web/src/routes/S2OrganismView.tsx:58`, `apps/web/src/routes/Dashboard.tsx:32`).
- **Render the shared-knowledge graph (KB)** — a second React Flow surface that draws the backend `ResearchNote` projection (the agents' research folded from the log) as generation-column / agenome-hub / note-leaf, with stigmergy `retrieved` edges and a graveyard for culled lineages (`apps/web/src/knowledge/KnowledgeGraph.tsx:59`).
- **Consume projections read-only over a typed seam** — a REST client (8 GET projections + a few demo reads) + an SSE stream, each Zod-validated on read (`apps/web/src/data/runClient.ts:139`, `apps/web/src/data/sseStream.ts:71`).
- **Fold events deterministically by `sequence`** — a pure run-store reducer + pure event-derived panel selectors, idempotent and replay-identical (`apps/web/src/state/reducer.ts:67`).
- **Issue the two idempotent commands** — start (`POST /runs`) and stop (`POST /runs/:id/stop`), with terminal/disabled state derived from folded events, never guessed (`apps/web/src/components/run/runControl.ts:71`).
- **Stay accessible + projector-legible** — status = shape+icon+label+color; a dark / high-contrast / light theme toggle (`apps/web/src/components/app/useTheme.ts:4`).

What it is **NOT** accountable for:
- It does **not** mutate or own authoritative state — read-only over projections; SSE is delivery-only (safety rule #2, `apps/web/src/data/sseStream.ts:4`).
- It does **not** re-judge, re-rank, or recompute any score — every score/critic/check/judge value is displayed **verbatim** (safety rule #6, `apps/web/src/panels/finalIdeaData.ts:19`).
- It does **not** enforce caps — the cap-clamp on the launcher form is fail-closed *UX only*; the kernel/route is the sole authority (rule #1, `apps/web/src/components/run/runConfigForm.ts:4`).
- It does **not** import `apps/api/**` — it shares only `packages/contracts` types (rule, `apps/web/src/data/contracts.ts:1`).
- It does **not** fetch or render any provider key/secret (safety rule #4).
- It does **not** build external `<a href>` links to traces — evidence refs render in-tier (rule #9, `apps/web/src/panels/evidenceRef.tsx:5`).

## Interfaces & contracts

**Inward — what it consumes (read-only).** The dashboard talks to exactly the `ARCHITECTURE.md` §11 endpoint set (plus the web-local `/runs/:id/knowledge`) through one client (`apps/web/src/data/runClient.ts:49`). No arbitrary URL/method is representable — the surface is hard-coded:

```
GET  /runs                       → RunSummary[]            (web-local)
GET  /runs/:id                   → RunStateView            (web-local, permissive `state`)
GET  /runs/:id/events?since=N    → RunEventEnvelope[]      (frozen contract)
GET  /runs/:id/lineage           → LineageGraphProjection  (frozen contract)
GET  /runs/:id/replay            → RunStateView
GET  /runs/:id/candidates/:cid   → CandidateIdea           (frozen contract)
GET  /model-routes               → ModelRoute[]            (frozen contract)
GET  /runs/:id/health            → RunHealth               (web-local; no frozen contract yet)
GET  /runs/:id/knowledge         → KnowledgeGraph          (web-local; the ResearchNote KB projection)
GET  /problem-sets               → ProblemSet[]            (web-local demo)
GET  /demo/fallback-ladder       → RungDescriptor[]        (web-local demo)
GET  /config/caps                → RunCaps                 (frozen, served read-only)
POST /runs           (start)     → StartRunResult          (idempotent; optional Idempotency-Key)
POST /runs/:id/stop  (stop)      → StopRunResult           (idempotent)
```

`getKnowledge` is declared on the `RunClient` interface (`apps/web/src/data/runClient.ts:69`) and implemented at `:199` — a plain read-only `GET /runs/:id/knowledge` parsed through the web-local `KnowledgeGraph` schema.

**The frozen-contract seam.** Every frozen schema is imported through a single re-export file (`apps/web/src/data/contracts.ts:12`) — the dashboard never redefines an Appendix-A model. Each imported name is *both* the runtime Zod schema (used to validate-at-boundary) and its inferred TS type. So a server payload that drifts from the contract surfaces as a typed `PayloadValidationError` rather than corrupting view state (`apps/web/src/data/errors.ts:9`).

**Web-local schemas (deferred-promotion).** **Five** shapes have **no frozen contract yet** and are mirrored web-locally, forward-tolerant (extra fields allowed): `RunHealth` (`apps/web/src/data/health.ts:14`), `ProblemSet` (`apps/web/src/data/operatorPromptClient.ts:15`), `RungDescriptor` (`apps/web/src/data/fallbackLadderClient.ts:27`), the REST envelope wrappers (`RunSummary`, `RunStateView`, command results — `apps/web/src/data/runClient.ts:108`), and now the KB `KnowledgeGraph` (`apps/web/src/data/knowledge.ts:42` — `{runId, sequenceThrough, state:{notes, edges, agenomes?}}`, with its `ResearchNote`/`ResearchEdge`/`ResearchAgenome` sub-shapes). Each carries an explicit **integration carry-forward** to reconcile/promote at the demo→cody merge — these are flagged deferrals, not shipped contract surface. The KB schema's `agenomes` field is **optional** so an older API without the graveyard fold degrades gracefully (no culled marks) rather than rejecting.

**Outward — what it exposes to its own routes.** A single app-level `runClient` via React context (`useRunClient` throws outside the provider — no silent null, `apps/web/src/data/RunClientProvider.tsx:23`); the `useRunObservatory` hook (live store + fold + projections, `apps/web/src/routes/useRunObservatory.ts:42`); and the DS component barrel (`apps/web/src/components/ds/index.ts:1`).

## Data & state

State lives in the **client**, all of it derived from the backend event log — nothing here is authoritative.

**`FoldState` (raw events).** Ordered, deduped `RunEventEnvelope[]` + a `lastSequence` watermark (`apps/web/src/data/sseStream.ts:23`). This is the panels' source — the chart/critic/energy/judge selectors all fold over `fold.events`.

**`ViewState` (the run store).** A *lean* derived view — `{ runId, lastSequence, entities, failures }` (`apps/web/src/state/reducer.ts:23`). It holds **per-entity latest status only** (run / generation / agenome / candidate keyed by most-specific id) plus a retained list of the 7 failure/terminal event types (never dropped — partial evidence, `apps/web/src/state/reducer.ts:38`). It deliberately does **not** hold metric values — panels read those from the raw events instead.

**Why two folds.** The store is lean and mode-agnostic (it never folds the `live|replay` mode flag); the panels need the full raw event list, so the shell keeps a *separate* `FoldState` alongside the store (`apps/web/src/routes/useRunObservatory.ts:73`). Both fold by `sequence`, so they agree.

**Projections (re-fetched, not folded).** `LineageGraphProjection`, `RunHealth`, and the KB `KnowledgeGraph` are rebuilt-on-read by the API, so a one-time fetch goes stale — the observatory hook re-fetches lineage/health on the SSE cadence (trailing debounce + forced-immediate on a terminal event) while run *state* stays live via the store fold (`apps/web/src/routes/useRunObservatory.ts:80`); the KB graph re-fetches on its own wall-clock poll (`apps/web/src/routes/KnowledgeView.tsx:62`).

**Key derived structures:** `FlowGraph` (RF nodes/edges, `apps/web/src/lineage/lineageToFlow.ts:77`); `InFlightState` (working entity ids + activity feed, `apps/web/src/lineage/inFlight.ts:68`); `FitnessSeriesPoint` / `GenerationComparisonPoint` (`apps/web/src/charts/chartData.ts:15`); `WinnerProof` (`apps/web/src/panels/finalIdeaData.ts:46`); `AgenomeTelemetry` / `JudgeRationaleView` (deep inspector, `apps/web/src/panels/nodeTelemetry.ts:40`,`:88`). Theme is a class on `document.documentElement` + `localStorage['doppl-theme']` (`apps/web/src/components/app/useTheme.ts:30`).

**Lineage node data gained two fields (FV.5a redesign).** `LineageNodeData` now carries `generationIndex?` (the zero-based generation ordinal — the layout's column key, passed through from the projection's new optional `LineageNode.generationIndex`) and `bornBy?` (`'seed' | 'fusion' | 'mutation'`, derived for agenome nodes from the *first* incoming reproduction edge, drives the body color-code) (`apps/web/src/lineage/lineageToFlow.ts:31`,`:65`).

**KB-graph derived structures.** `knowledgeToFlow(state)` produces `{nodes, edges}` where each node carries `KnowledgeNodeData` — `kind ∈ {generation, agenome, note}`, the `generationIndex` column key, note fields (`toolName`/`query`/`snippet`/`sourceUrls`), the hub `noteCount`, and the graveyard `culled`/`score` (`apps/web/src/knowledge/knowledgeToFlow.ts:17`). Each `KnowledgeRfEdge` carries `data.edgeType ∈ {spawned (synthesized backbone), researched, cited, retrieved}`; a `retrieved` edge also gets `animated:true`. `KnowledgeGraph` (the component) keeps a `shown` state that tracks the freshest projection by `sequenceThrough` watermark (`apps/web/src/knowledge/KnowledgeGraph.tsx:55`) — the same stale-watermark guard the lineage graph uses. The deterministic columns (`apps/web/src/knowledge/layout.ts:21`) use `COL_WIDTH=320 / ROW_HEIGHT=116` (the lineage layout uses `340 / 132`) — both pure, sort by id, no RNG/wall-clock.

## Dependencies

- **Depends on:** `packages/contracts` — the frozen Zod schemas (`RunEventEnvelope`, `CandidateIdea`, `LineageGraphProjection`, `FitnessScore`, `NoveltyScore`, `EnergyEvent`, `CriticReview`, `CheckResult`, `JudgeResult`, `LlmCallTelemetry`, `ModelRoute`, `RunConfig`/`RunCaps`) — consumed read-only via `apps/web/src/data/contracts.ts:12`. **Types only**; it imports no `apps/api` internals. (The KB `KnowledgeGraph` shape is web-local, NOT a frozen contract — see above.)
- **Depends on:** the **backend API + SSE** (`07-backend-api-rest-sse.md`) at runtime — REST projections (incl. the new `GET /runs/:id/knowledge`) + the run-event stream, reached in local dev via a Vite proxy `/api`→`http://localhost:3000` (base overridable via `VITE_API_BASE`, `apps/web/src/data/apiBase.ts:12`).
- **Depends on:** the **projections** (`06-projections-read-models.md`) for the actual view shapes it renders (lineage graph, current-state, health, replay summary) and the **shared knowledge space** (`11-shared-knowledge-space.md`) for the `ResearchNote` KB projection the Knowledge-Evolution graph draws.
- **Used by:** the human operator / demo audience — this is the terminal consumer; nothing in the codebase calls *into* the dashboard.

## How it works (flow)

```
 [REST projections] ─┐
                     ├─► useRunObservatory ─► createRunStore.applyEvent ─► ViewState ─► useSyncExternalStore ─► panels
 [SSE run events]  ──┘        │                  (lean fold by sequence)
   (wireRunStream)           ├─► raw FoldState (events) ─► chart/critic/energy/judge selectors ─► panels
                             └─► debounced re-fetch ─► getLineage / getRunHealth ─► LineageGraph / HealthIndicator

 [GET /runs/:id/knowledge] ─► KnowledgeView (4s poll) ─► KnowledgeGraph (freshest by sequenceThrough)
                                                          └─► knowledgeToFlow + layoutKnowledge ─► React Flow
```

1. **Boot + route.** `App` memoizes one `runClient` (baseUrl from `import.meta.env.VITE_API_BASE ?? '/api'`) and hands it to every route via context; `AppRoutes` maps URL → screen behind a layout-route `AppShell`, with the observed run + mode URL-derived (`apps/web/src/App.tsx:15`, `apps/web/src/app/routes.tsx:63`).
2. **Wire the live data.** On a run screen, `useRunObservatory` creates a `RunStore`, seeds the raw `FoldState` from `getEvents`, fetches the initial lineage/health, then calls `wireRunStream` — which resyncs the store from REST, opens the SSE stream, and routes each envelope to `store.applyEvent` (the single ViewState sink) **and** the raw-events accumulator (`apps/web/src/routes/useRunObservatory.ts:80`, `apps/web/src/routes/dashboardWiring.ts:29`).
3. **Fold by sequence.** Both folds accept an envelope iff `sequence > lastSequence`; `occurredAt` is never consulted, so a duplicate or a late lower sequence is dropped and the fold is idempotent (`apps/web/src/data/sseStream.ts:35`, `apps/web/src/state/reducer.ts:67`).
4. **Re-fetch projections live.** Each non-terminal envelope schedules a debounced lineage+health re-fetch; a terminal envelope (`run.completed/failed/stopped`) cancels the debounce and forces an immediate final fetch so the FINAL graph always renders (`apps/web/src/routes/useRunObservatory.ts:113`).
5. **Render the graph.** `LineageGraph` keeps the freshest projection by `sequenceThrough`, maps it via `lineageToFlow` (declutter to the agenome+candidate backbone; detail moves to the inspector; nodes color-coded by `bornBy`, edges styled per type), lays it out with the **per-generation column layout**, and overlays the in-flight "working…" state from `deriveInFlight` over the raw events; a fixed `LineageLegend` is pinned top-right (`apps/web/src/lineage/LineageGraph.tsx:104`).
6. **Click a node → inspector.** `onNodeClick` sets the selected node; `NodeInspectorContent` routes a candidate to the existing panels (inspector + fitness + critic gauntlet + subtype checks + judge rationale) and an agenome to its deep generation telemetry — all pure over the same events (`apps/web/src/components/run/NodeInspectorContent.tsx:69`).
7. **Replay.** In replay mode the screen mounts a step scrubber; `foldAtStep(events, n)` re-folds `events[0..n)` client-side via the same pure `foldEvents` — no server call, no provider call (`apps/web/src/routes/replayScrubber.ts:16`).
8. **Commands.** Start/stop go through the REST client; the stop control derives terminal/disabled state from the folded run-level event, never optimistically (`apps/web/src/components/run/runControl.ts:71`).

**Knowledge-Evolution graph path (separate from the observatory hook).** A user on a run screen clicks **Knowledge** in the `AppShell` run-nav (`apps/web/src/components/app/AppShell.tsx:67`) → the router matches `/runs/:id/knowledge` to `KnowledgeRoute`, keyed by id (`apps/web/src/app/routes.tsx:71`, `:40`) → `KnowledgeView` mounts and its effect calls `runClient.getKnowledge(runId)` immediately, then on a 4s interval (`apps/web/src/routes/KnowledgeView.tsx:62`). The API answers from `buildResearchNotes` folding the event log (`apps/api/src/routes/runs-read.ts:80`), the client validates through the web-local `KnowledgeGraph` Zod schema, `KnowledgeGraph` keeps the freshest projection by `sequenceThrough`, and `knowledgeToFlow` + `layoutKnowledge` produce the generation-column / agenome-hub / note-leaf graph (with the synthesized `spawned` backbone and the projection's `researched`/`retrieved` edges; `retrieved` rendered as an animated cyan-dashed stigmergy trail). `KnowledgeLegend` is pinned top-right.

## Design decisions & rationale

- **Multi-route app over a single mount (FV.1, ARCHITECTURE.md §12).** The dashboard is `react-router-dom` behind a global `AppShell` layout route — routes `/` (S0 home), `/launch` (S1), `/runs/:id[/replay]` (S2 organism), `/runs/:id/knowledge` (Knowledge-Evolution graph), `/runs/:id/final` (S5). The observed run + mode are URL-derived (a per-`(mode,id)` `key` remount tracks the address bar so back/forward/bookmark work — `apps/web/src/app/routes.tsx:36`). The `AppShell` surfaces a contextual **Organism / Knowledge** nav whenever a `/runs/:id/*` sub-route is active (`apps/web/src/components/app/AppShell.tsx:56`), absent on home/launch. The data layer is route-agnostic: one app-level `runClient` via context, the store/SSE wiring extracted into `useRunObservatory` so S2 and S5 reuse it intact.
- **Two folds, lean store.** The store stays minimal (status + failures) and mode-agnostic so live and replay fold identically; the metric-bearing panels read raw events. This is the project's banked lesson (apps/web LESSONS §2/§6) — the store is a pub-sub *and* the SSE sink, decoupled from stream construction for testability.
- **Pure selectors everywhere.** Charts, critic/check/energy panels, the final-idea proof, and the deep telemetry are all pure functions over the persisted events, validated through the frozen Zod and read verbatim. This is what makes **replay byte-identical** to live (rule #7) and keeps scoring authoritative (rule #6).
- **Declutter the graph, keep the projection complete (FV.5a).** `lineageToFlow` drops `critic`/`check`/`score` leaf nodes to the agenome→candidate backbone and moves their detail to the node-click inspector; the underlying projection stays complete/authoritative — the declutter is presentation-only (`apps/web/src/lineage/lineageToFlow.ts:107`).
- **Deterministic per-generation column layout (FV.5a redesign; was Dagre).** The lineage graph originally used `@dagrejs/dagre`'s network-simplex auto-layout. That was **dropped** because a non-expert couldn't read which organisms belong to which generation. `layoutGraph` now buckets every node into a fixed **column = its `generationIndex`** (the generation header chip on top, then each agenome with its candidate(s) stacked beneath), pure + RNG/clock-free, so the same projection lays out identically every render and the graph reads left→right as the run advances (`apps/web/src/lineage/layout.ts:23`). The knowledge graph uses the same column scheme (`apps/web/src/knowledge/layout.ts:21`). This preserves the same determinism guarantee Dagre gave (required for a stable projector view) while making the evolution legible. **DRIFT (stale source residue):** an unused `@dagrejs/dagre` dependency still sits in `apps/web/package.json:19` (no source imports it), and two docstrings still mention Dagre (`apps/web/src/lineage/LineageGraph.tsx:16`, `apps/web/src/lineage/layout.ts:5`); behavior is the column layout — flagged, not fixed.
- **Read evolution at a glance — operation color-coding + per-type edge visuals + fixed legends (FV.5a redesign).** Lineage nodes are color-coded by operation: a winner glows gold, a culled lineage gets a stronger red wash + outline, and an agenome is tinted by its `bornBy` hue (seed grey-blue / mutation amber / fusion violet) — never by color alone, the `StatusBadge` glyph+label stay (`apps/web/src/lineage/nodeTypes.tsx:86`). Each edge is styled per type by `edgeStyleFor` — fusion family violet-thick-animated, `mutation_only` amber-dashed-animated, `generated`/`spawned` faint — with `smoothstep` routing and **no** per-edge text labels (the legend carries the key instead, decluttering the canvas) (`apps/web/src/lineage/edgeStyles.ts:41`, `apps/web/src/lineage/lineageToFlow.ts:156`). Both graphs pin a fixed top-right legend (`apps/web/src/lineage/LineageLegend.tsx:84`, `apps/web/src/knowledge/KnowledgeLegend.tsx:35`). The `--status-culled` / `--edge-culled` tokens were recolored **gray → deep "sunk" red** (`apps/web/src/styles/tokens/colors.css:50`,`:93` dark; `:155`,`:193` light) so a non-expert SEES selection killing weak lineages.
- **A second React Flow surface: the Knowledge-Evolution graph (KB).** Distinct from the lineage/organism graph, it renders the backend `ResearchNote` projection — the swarm's accreting *shared research* (stigmergy) — as generation-column / agenome-hub / note-leaf, read left→right as the KB grows. Notes are color-coded by the research **tool** that produced them (web / X / YouTube / fetch), each with a glyph + label; the in-run `retrieved` edge (an agent following a prior agent's research) is an **animated cyan-dashed** stigmergy trail, visually distinct from the solid `researched` WRITE edge. It is strictly read-only and pure over the projection (rules #2/#7) (`apps/web/src/knowledge/knowledgeToFlow.ts:73`).
- **Viewport-clamped cockpit + stick-to-bottom ticker.** The S2 organism view is clamped to `calc(100vh − header)` with three independently-scrolling panes (`min-height:0` grid tracks) so streaming content never grows the page (`apps/web/src/routes/S2OrganismView.tsx:58`); the `ActivityTicker` flips to **ascending** (newest at the bottom) with a pure `isAtBottom` stick-to-bottom predicate, a "Jump to latest" affordance, and a soft 500-row DOM cap (`apps/web/src/components/ds/ActivityTicker.tsx:81`). The (now-unrouted) `Dashboard`'s ~10 operator panels flow into an auto-fit responsive grid (`apps/web/src/routes/Dashboard.tsx:89`).
- **Accessibility is load-bearing, not cosmetic (§12).** Status is shape+icon+label+color (the color is the 4th, redundant channel); the glyph is `aria-hidden`, the status reaches assistive tech via the label/`title`; a dark/high-contrast/light theme. Both legends and the new tool/edge encodings follow the same rule (color never the sole channel). The dashboard is "a first-class acceptance surface shown to a room."
- **Web-local schemas with explicit promotion carry-forwards.** P0 (the contracts package) is frozen, so the demo track can't add Appendix-A models unilaterally — health/problem-set/ladder/envelope-wrapper/KB-graph shapes are mirrored web-locally and flagged for reconciliation at merge (documented deferral, not silent drift).

## Safety & invariants

This layer is the *consumer* boundary; its safety posture is enforced **structurally** (by what the code can and cannot represent), not by prose.

- **Safety rule #2 — UI never mutates authoritative state; SSE is delivery-only.** The REST client exposes exactly the contract endpoints (plus the read-only KB `getKnowledge`), and *only* two of them write — `startRun` / `stopRun`, both idempotent (`apps/web/src/data/runClient.ts:49`). The SSE stream is a pure fold sink with no write path; ordering is by `sequence` alone (`apps/web/src/data/sseStream.ts:35`). Dropping the stream loses no state — `resyncFromRest` re-fetches from the last applied `sequence` (`lastEventId`) and folds to the identical view (`apps/web/src/state/resync.ts:22`), and the store's `onError` falls back to `poll()` (`apps/web/src/routes/dashboardWiring.ts:40`). The stop control's terminal state is *derived from the folded `run.stopped/completed/failed` event*, never guessed optimistically (`apps/web/src/components/run/runControl.ts:71`). The KB graph is the same read-only posture: `KnowledgeView` issues no commands and `KnowledgeGraph` keeps the freshest projection by `sequenceThrough` so a stale poll never regresses the view (`apps/web/src/knowledge/KnowledgeGraph.tsx:55`).
- **Safety rule #4 — no secret in the client.** There is no code path that fetches or renders a provider key: the client surface is the projection/command set only, and the Dashboard's "Secret redaction active" indicator is a *static* trust note, not a value (`apps/web/src/routes/Dashboard.tsx:247`). The deep telemetry inspector and the KB note cards render only the *already-scrubbed, truncated-at-persistence* captures — they add no new secret surface (`apps/web/src/panels/nodeTelemetry.ts:12`, `apps/web/src/knowledge/nodeTypes.tsx:117`).
- **Safety rule #6 — judge/scoring immutable, emit-only display.** Every score/critic/check/judge value is read **verbatim** through the frozen Zod and displayed; nothing is recomputed or re-ranked. The final-idea panel shows the kernel-marked `'selected'` winner and ignores a higher-metric non-selected candidate (`apps/web/src/panels/finalIdeaData.ts:19`,`:26`); the chart selectors read `total`/`score` verbatim (`apps/web/src/charts/chartData.ts:8`); the judge-rationale selector displays the explanation but derives no acceptance (`apps/web/src/panels/nodeTelemetry.ts:101`). The KB graveyard shows each culled lineage's cull `score` verbatim — it re-ranks nothing.
- **Layer rule — no backend internals.** Frozen schemas are imported only via `apps/web/src/data/contracts.ts:12` from `@doppl/contracts`; the forbidden-pattern grep (`from ['"].*apps/api/`) backs this. The real web→API smoke runs the API as a *child process*, never an import (`apps/web/test/smoke/web-api-smoke.test.ts:14`).
- **Rule #7 (replay) + #8 (energy) reflected.** Replay re-folds persisted events client-side with no provider call (`apps/web/src/routes/replayScrubber.ts:8`); the energy panel counts only `energy.spent` and reads `EnergyEvent.actual` verbatim — failed/retried attempts add nothing, and the exhaustion decision is read from the kernel's `energy_exhausted` event, never a client-side `total≥budget` compare (`apps/web/src/panels/energyData.ts:4`). `knowledgeToFlow`/`layoutKnowledge` are pure (no RNG/wall-clock; sort by id), and the `retrieved` edges are folded server-side from persisted `candidate.generation_started` markers — so a replay renders an identical KB graph with no provider call (`apps/web/src/knowledge/layout.ts:21`).
- **Rule #9 — in-tier evidence refs.** `EvidenceRefLink` renders eventId/uri/observationId as text + `data-*` attributes for the shell to resolve within the Postgres tier — it never constructs an external `<a href>` (`apps/web/src/panels/evidenceRef.tsx:5`); its test asserts no `<a>`/`[href]`.
- **Rule #1 reflected (not enforced).** The launcher clamps caps to the fetched `/config/caps` maxima — but this is fail-closed *UX only*; the kernel/route is the sole enforcer (`apps/web/src/components/run/runConfigForm.ts:4`).

## Gotchas & sharp edges

- **The lean store drops metrics.** `ViewState` holds only per-entity *status*, not metric values — a panel reading from the store instead of the raw `fold.events` would see nothing. Charts/energy/critic panels must fold `fold.events` (apps/web LESSONS §2/§6).
- **Two re-fetch cadences, one stale trap.** Run *state* is live via the store fold, but lineage/health are *projections* rebuilt-on-read — folding the raw feed is **not** the same as rebuilding the projection. A one-time fetch froze the graph at 1 node until the debounced re-fetch was added; a terminal envelope must force the final fetch or the FINAL graph can be lost to debounce coalescing (`apps/web/src/routes/useRunObservatory.ts:113`, apps/web LESSONS §13).
- **`KnowledgeView` polls on a wall-clock interval, NOT the SSE cadence.** It does **not** share `useRunObservatory`; it re-fetches `getKnowledge` every 4s (rebuild-on-read, injectable; `0` disables for tests). The freshest-by-`sequenceThrough` watermark guard inside `KnowledgeGraph` (`apps/web/src/knowledge/KnowledgeGraph.tsx:55`) is what makes the poll safe against an out-of-order/stale response. `KnowledgeView` also does not pass `onNodeClick`, so a node click is a no-op at the route level today (`apps/web/src/routes/KnowledgeView.tsx:54`).
- **DRIFT (CONFIRMED) — `cited` knowledge edges are effectively never rendered.** The backend emits `cited` edges with `source = candidateId, target = noteId` (`apps/api/src/projections/research-notes.ts:231`), but `knowledgeToFlow` builds nodes ONLY for generations / agenomes / notes (no candidate nodes), so every `cited` edge is dropped as dangling at `apps/web/src/knowledge/knowledgeToFlow.ts:161`. The gold `cited` branch in `knowledgeEdgeStyle` (`:64`) is therefore **unreachable** for the current node set, and `KnowledgeLegend` correctly omits a `cited` row. This is forward-looking/defensive (a clean mirror of the producer — RF breaks on dangling edges), not a runtime bug — but the cited stigmergy relationship is invisible in the KB graph today even though the API folds it. `researched` and `retrieved` (both agenome→note) DO render.
- **The KB summary header counts agents/culled over researching agenomes only.** `agents` = distinct `agenomeId`s that PRODUCED a note; `culled` = agenomes that are both culled AND researching (`apps/web/src/knowledge/KnowledgeGraph.tsx:72`). A culled lineage that did no research has an `agenomes` entry but no hub and is excluded, so the header can never read "1 agents · 2 culled."
- **YouTube notes are cyan, deliberately NOT red.** `youtube_search` notes use `--status-active` (cyan); red (`--status-culled`) is reserved exclusively for the graveyard (dead-end) treatment, so a live YouTube note is never visually mistaken for culled research (`apps/web/src/knowledge/nodeTypes.tsx:18`).
- **S2 cockpit hard-codes the header height as a token calc.** `APP_HEADER_H = calc(--space-3 + --space-3 + --text-h3-lh)` (`apps/web/src/routes/S2OrganismView.tsx:58`) — if the AppShell header padding or wordmark line-height changes, this calc must track it or the panes mis-clamp.
- **`ActivityTicker` auto-scroll needs a real layout engine.** The stick-to-bottom rule was extracted into the pure `isAtBottom` predicate so it stays unit-testable under jsdom (which has no layout) (`apps/web/src/components/ds/ActivityTicker.tsx:81`).
- **Effect-dep churn footgun.** Inline default factories (e.g. `eventSourceFactory = url => new EventSource(url)`) are a *new function each render* → the wiring effect re-runs → a re-fetch loop that cancels state before commit. Both `Dashboard` and `useRunObservatory` use a **module-const** default to stabilize the dep (`apps/web/src/routes/useRunObservatory.ts:54`; the real e2e smoke caught this, apps/web LESSONS §10).
- **React Flow breaks on dangling edges.** Both `lineageToFlow` and `knowledgeToFlow` drop any edge whose source/target endpoint is missing (or was decluttered) — a defensive mirror of the producer (`apps/web/src/lineage/lineageToFlow.ts:156`, `apps/web/src/knowledge/knowledgeToFlow.ts:161`).
- **The replay scrubber rewinds the *overlay*, not the node set.** `foldAtStep` rewinds the in-flight overlay + the fold-derived panels, but the lineage **node structure stays full** (the API-projected graph isn't reconstructed per-step). This is a flagged limitation, not a bug (`apps/web/src/routes/S2OrganismView.tsx:107`, apps/web LESSONS §22).
- **DRIFT (harmless, stale prose) — `contracts.ts` "health absent" note.** `apps/web/src/data/contracts.ts:8` says "GET /runs/:id/health is intentionally absent … deferred to P7.14." In the shipped code health **is** wired — via the web-local `RunHealth` schema (`apps/web/src/data/health.ts:14`) and `runClient.getRunHealth` (`apps/web/src/data/runClient.ts:198`). The comment is stale; behavior is correct.
- **DRIFT (carry-forward) — judge in-flight under schemaVersion 2.** `deriveInFlight` notes `judge.review_started` has no paired `judge.reviewed` completion in sv2, so a judge marker stays "working" forever under sv2 (`apps/web/src/lineage/inFlight.ts:12`). Documented; folds into the sv3 demo→cody reconcile. The deep judge-rationale selector consumes `judge.reviewed`/`JudgeResult` which is sv3 (`apps/web/src/panels/nodeTelemetry.ts:101`).
- **DRIFT (interim → superseded) — AppShell ModeBanner slot + dual observatory.** `AppShell` still reserves an (empty) ModeBanner slot "until FV.4 lifts it here" (`apps/web/src/components/app/AppShell.tsx:78`); the banner currently renders inside each screen. The shell now ALSO renders the contextual Organism/Knowledge run-nav (`apps/web/src/components/app/AppShell.tsx:67`). Separately, the monolithic `Dashboard.tsx` keeps its **own inline copy** of the observatory wiring (a *tracked temporary duplication* of `useRunObservatory`, `apps/web/src/routes/Dashboard.tsx:172`); the live screens (S2/S5) use the hook. `Dashboard` is **effectively unrouted** in `AppRoutes` (`apps/web/src/app/routes.tsx:66-74` mounts S0/S1/S2/Knowledge/S5, not Dashboard) — so its new responsive panel grid (`apps/web/src/routes/Dashboard.tsx:89`) is a refinement to a retained-for-tests / pre-router-fallback component. **UNVERIFIED:** whether `Dashboard.tsx` is reachable from any route in the shipped build — by inspection of `routes.tsx` it is not.
- **UNVERIFIED — connection-drop SSE fallback.** `createSseStream`'s `onError` fires on a *payload-validation* failure; a connection-*drop* fallback (an EventSource `'error'` listener) is not modeled by `EventSourceLike` and is flagged for the real-EventSource wiring at merge (`apps/web/src/routes/dashboardWiring.ts:14`).
